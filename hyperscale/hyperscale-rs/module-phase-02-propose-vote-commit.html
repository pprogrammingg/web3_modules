0 — Prerequisite mental model

  • One BFT instance per shard — validators in the committee run the same protocol locally; progress requires quorum (typically 2f+1 of n=3f+1).
  • Height = block slot index; round (view) = attempt number at that height. Round advances on view-change timeout, not on a “proposal timer.”
  • Proposals are event-driven, not clock-driven. Hyperscale-rs does not fire a periodic “time to propose” alarm. Instead, whenever something actually changes what could go into the next block—a new mempool transaction, a learned QC, completed cross-shard provisions, or a fresh commit—the node sets a lightweight latch with queue_ready_proposal(): “inputs may have changed; check whether we should build.” After the current NodeInput is handled and its actions are dispatched, try_event_driven_proposal gathers ready material and, if this validator is the leader for the current height and round, calls try_propose. Rounds still advance on view-change timeout when the protocol stalls—but there is no ProposalTimer in the codebase today.
  • Quorum certificates ride the chain, not a separate wire. When enough validators agree on block H, they produce a quorum certificate—QC(H). You might expect a dedicated “here is QC(H)” gossip message; in practice, hyperscale-rs mostly does not work that way. The next proposer carries QC(H) inside block H+1’s header as parent_qc, so the certificate spreads with normal block gossip. If you aggregated votes yourself, you may already hold QC(H) before H+1 lands—but if you are tracing “where did everyone learn QC(H)?”, follow parent_qc on the child block, not a standalone QC broadcast.

1 — Proposer selection (step 7)

Leader is deterministic from topology — not voted per block:

  • topology.proposer_for(height, round) — round-robin over the shard’s ordered committee list: committee[(height + round) % committee.len()]. The array slot is a ValidatorId; every honest node derives the same proposer from height and round alone.
  • topology.should_propose(height, round) — true only when that returned id equals this node’s local_validator_id (only the designated proposer dispatches BuildProposal).
Gotcha — small committees: With 5 validators, the same node is proposer again every 5 heights at round 0. That is expected round-robin, not “unfair repetition.”
Gotcha — Byzantine header: Admission checks proposer_for(height, header.round()) matches the header’s claimed proposer; wrong proposer → reject before voting.

In-repo proof — round-robin proposer; local node proposes only when ids match

// crates/types/src/topology/snapshot.rs
pub fn proposer_for(&self, height: BlockHeight, round: Round) -> ValidatorId {
    let index = (height.inner() + round.inner()) % committee.len() as u64;
    committee[index]
}
pub fn should_propose(&self, height: BlockHeight, round: Round) -> bool {
    self.proposer_for(height, round) == self.local_validator_id
}

2 — Latch: when does the node try to propose?

Proof snippets below are trimmed from the hyperscale-rs tree. Ellipsis (...) hides boilerplate; follow data-file links for full sources.

A

queue_ready_proposal()

Called when inputs change: txs admitted, QC formed, genesis/recovery, provisions, etc. Sets a latch — “something worth proposing may exist.”

B

try_event_driven_proposal

After NodeStateMachine::handle finishes an event, node gathers ready_transactions(), finalized waves, provisions → calls bft.try_propose(...) (proposal.rs).

C

try_propose guards

should_propose, parent tree available, pending-proposal latch (no duplicate BuildProposal for same height/round), backpressure. May return empty (defer) or Action::BuildProposal.

This phase is the canonical BFT coordinator reading—routine names, in-repo proofs, and staff judgment about timing and failure modes.

3 — Build block (step 8)

  1. Action::BuildProposal — IoLoop runner executes (actions.rs).
  2. action_handlers::build_proposal — reads storage (JMT), assembles BlockHeader + body from candidate txs, waves, certs (action_handlers.rs). BftCoordinator does not build bytes here.
  3. ProtocolEvent::ProposalBuilt — fed back into state machine.
  4. on_proposal_built — logs “Broadcasting proposal”, stores PendingBlock, returns BroadcastBlockHeader + own BlockVote actions.
Gotcha — stale ProposalBuilt: A pending-proposal latch ties callbacks to the dispatched BuildProposal. Late storage callbacks for an old height/round are dropped — prevents double-broadcast from repeated try_propose.

In-repo proof — BFT emits Action::BuildProposal; runner/storage builds bytes (not in coordinator)

// crates/bft/src/coordinator.rs — try_propose returns actions like:
return self.build_and_dispatch_proposal(
    topology_snapshot, next_height, round, ProposalKind::Normal,
);
// → Action::BuildProposal { height, round, parent_qc, transactions, ... }

4 — Peers receive header (still step 8)

  • ProtocolEvent::BlockHeaderReceived + manifest → verification pipeline (roots, provision roots, timestamp bounds).
  • Validators that accept the header vote via PersistAndBroadcastVote (BLS committee signatures — distinct from user Ed25519 in Phase 1).
  • Incomplete blocks sit in PendingBlock until txs/certs arrive (fetch timeouts on cleanup timer — operational, not proposal timer).

In-repo proof — user Ed25519 was Phase 1; header votes use validator BLS (see Phase 2 Q&A)

// crates/core/src/action.rs (variant)
PersistAndBroadcastVote { vote: BlockVote, ... }
// Emitted after header/manifest verification accepts the proposal

5 — Voting (step 9)

  • Votes gossip as BlockVote notifications (class: round-blocking for consensus — delay can stall the round until view change).
  • VoteSet collects votes per block hash; when quorum threshold met → Action::VerifyAndBuildQuorumCertificate.
  • Crypto verification often offloaded; result → ProtocolEvent::QuorumCertificateFormed or QuorumCertificateResult.

Equivocation vs revote?

Equivocation = conflicting votes for the same height and round. Revote = voting again after a new round at the same height (leader timed out, new proposer) — allowed.

In-repo proof — quorum triggers off-thread QC build

// crates/core/src/action.rs
VerifyAndBuildQuorumCertificate { block_hash, votes, ... }
// Runner completes → ProtocolEvent::QuorumCertificateFormed

6 — QC formed (step 10)

  1. action_handlers::verify_and_build_qc aggregates BLS signatures into QuorumCertificate.
  2. BftCoordinator::on_qc_formed updates latest_qc, may emit BlockReadyToCommit when two-chain condition satisfied.
  3. QC propagation: Next block’s parent_qc carries QC(H); aggregators may learn QC earlier when they formed it locally.
Gotcha — “send the QC”: There is no required standalone QC gossip for liveness. Design assumes validators embed the certifying QC in the next proposal or learn via subsequent headers.

In-repo proof — on_qc_formed updates latest_qc; may chain into next try_propose

7 — Commit (step 11)

Two-chain rule (HotStuff-style): block at height H can commit when the node has observed a QC on H+1 (directly or via H+2’s parent_qc). Every honest validator in the shard runs the same commit check locally.

  • on_block_ready_to_commit → commit_block_and_buffered (coordinator.rs).
  • Action::CommitBlock — node orchestrates persistence, mempool, execution hooks (state/bft.rs).
  • ProtocolEvent::BlockCommitted — downstream coordinators react on the same turn or follow-up events.

In-repo proof — every validator runs commit locally on BlockCommitted

// crates/node/src/state/bft.rs — on_block_committed (order)
self.bft.on_block_committed_verification(block_hash);
actions.extend(self.mempool.on_block_committed(self.topology.snapshot(), certified));
actions.extend(self.remote_headers.on_block_committed(...));
actions.extend(self.provisions.on_block_committed(...));
// → Action::CommitBlock from BFT when two-chain rule fires

8 — Mempool status on commit (single shard)

On MempoolCoordinator::on_block_committed (mempool):

  • Txs in the block move from Pending toward Committed (then execution updates toward Completed — Phase 3).
  • Txs not included stay Pending (may be proposed in a later block).
  • current_ts (chain-anchored) advances from the committing QC’s weighted_timestamp — drives validity sweeps on cleanup timer.

Proximate: successful commit may call queue_ready_proposal() again for the next height.

In-repo proof — mempool + execution wired on same commit event

// crates/node/src/state/bft.rs (comment on mempool arm)
// Mempool: marks Pending → Committed for block.transactions,
// then drives each tx in block.certificates to terminal state.

// crates/execution/src/coordinator.rs — on_block_committed
self.committed_ts = certified.qc().weighted_timestamp();
actions.extend(self.check_exec_cert_timeouts(topology));

9 — Design Q&A (curated)

Why no periodic proposal timer?

Proposals are driven by new work (mempool ready set, new QC, provisions). Liveness when the leader stalls is ViewChangeTimer → round bump → new proposer_for — see timing module.

Can the proposer censor txs in the block body?

Yes at the mempool-selection layer (omit txs). Mitigation: view change rotates leader; users gossip to all validators (Phase 1); eventually another proposer includes the tx if it stays valid and ready.

Can a validator vote for two different blocks at the same height and round?

That is equivocation (safety violation if quorum builds on both). Protocol + ops assume BLS accountability and slashing narratives off-repo; honest code tracks one vote line per (height, round).

Does only the proposer commit?

No. Every validator commits when its local two-chain condition fires — proposer is just the one who built the block at H.

BLS batch verify on block headers?

Committed-header gossip uses batch BLS verify for sender attestation (CommittedBlockHeader) — related to cross-shard light-client pattern, not the user-tx Ed25519 path from Phase 1.

10 — Staff debate: pros, cons, mitigations

Event-driven propose

Work happens when there is content; no empty tick load.

Cost

Leader can stall without proposing while holding the round open until MAX_PROGRESS_WAIT / view change.

Mitigation: View-change timeout + implicit round advance.

QC in next header

Bandwidth-efficient; ties commit evidence to chain structure.

Cost

Harder to debug “where is QC(H)?” for newcomers.

Mitigation: Trace parent_qc on H+1; metrics on CommitSource (aggregator vs header).

Per-validator commit

No central committer; matches BFT replication model.

Cost

Sync lag → validator commits behind; execution/mempool see delayed current_ts.

Mitigation: Sync protocol + check_sync_health on cleanup timer.

11 — Phase 2 checklist

  • Proposer known: proposer_for(height, round); only that node dispatches BuildProposal
  • Block built off storage via build_proposal; header gossip + votes
  • QC formed at 2f+1; learned via aggregation and/or next parent_qc
  • Two-chain commit applied on every validator; BlockCommitted fired
  • Included txs left Pending → Committed path; others stay Pending
  • Not yet: execution waves complete, cross-shard certs → Phases 3 / 4
