Arbitrum Nitro architecture

Most important module for Arbitrum-style interviews. Nitro is not “another fast L1”—it is modified Geth + ArbOS + a proving pipeline inside an optimistic rollup that posts data and assertions to Ethereum. You should sound like you understand how Ethereum compatibility is preserved while execution is optimized—and what still depends on L1.

Segment 1 — What Nitro is (vs monolithic L1)

One-sentence architecture

Arbitrum Nitro runs an Ethereum-compatible state transition function (STF) offchain, posts compressed batch data to Ethereum for availability, and defends correct execution with permissionless interactive fraud proofs (BoLD)—while using go-ethereum (Geth) for EVM execution and a WASM-compiled copy of the STF for efficient dispute resolution.

Four design principles (official framing)

  1. Sequencing first, deterministic execution second — order is explicit; STF replays that order.
  2. Geth at the core — reuse battle-tested EVM semantics and state representation.
  3. Separate execution from proving — native speed for nodes; WASM for challengers/validators.
  4. Optimistic rollup with interactive proofs — assume validity until challenged; bisect to a single step verifiable on L1.

Three-layer software stack

Layer Component Role
Top Node interface (RPC, feeds) Ethereum-like APIs, sequencer feed, operator tooling
Middle ArbOS (Arbitrum OS) L2 gas pricing, L1 data charges, cross-chain messaging, batch parsing, Stylus hooks, chain params
Base Modified Geth EVM execution, state DB, receipts—minimal invasive hooks

ArbOS state lives inside Geth’s state database (not a separate mystery DB)—compatibility is intentional.

WASM execution & proving (not the same as Stylus-only)

  • The Nitro STF is compiled to native code for full nodes and to WASM so disputes can run in a machine-independent sandbox.
  • Stylus (optional feature) adds user WASM contracts with host I/O into chain state—10–70× cited speedups vs EVM for some workloads; fully interoperable with EVM calls per docs.
  • Interview clarity: “Nitro uses WASM for proving the STF; Stylus uses WASM for contracts—related but different.”

vs monolithic L1 (Hyperscale-style)

  • L1: consensus + execution + DA + governance often in one operational story.
  • Nitro: sequencer + batch poster + validator set + Ethereum contracts for inbox/outbox/assertions—security inherits Ethereum for data and dispute resolution, optimizes execution throughput and costs offchain.

Segment 2 — Sequencer, inbox, batching, L1 posting

Submission paths

  • Happy path — Sequencer RPC — low latency; txs appear on the sequencer feed with soft finality.
  • Censorship-resistant path — Delayed Inbox on L1 — user sends to an L1 inbox contract; txs queue for the sequencer (~10 minutes typical inclusion per docs); after 24 hours anyone can force inclusion—sequencer cannot permanently censor, only delay.

Ordering & broadcasting

  1. Sequencer orders txs and publishes the real-time feed (soft finality).
  2. Txs are grouped into batches (size or time bound) and compressed (Brotli; level 0–11 adapts to congestion).
  3. Batches post to L1 via Sequencer Inbox contract:
    • Preferred: EIP-4844 blob transactions when economical.
    • Fallback: calldata when blobs are expensive or unavailable.

Dual finality (memorize this)

  • Soft finality — included in sequencer feed; fast; not cryptographically final on L1.
  • Hard finality — batch committed on Ethereum; inherits L1 consensus; typically ~10–20 minutes depending on batching and L1 block times; tied to rollup assertions on L1.

Inbox mental model

Think Inbox → STF → Outputs (Arbitrum’s napkin architecture):

  • Inbox — ordered byte stream of inputs (sequencer batches + delayed inbox messages).
  • STF — deterministic transition to new state.
  • Outputs — roots and messages consumable on L1 (outbox, assertions).

Segment 3 — STF execution, optimistic rollup flow, fraud proofs (BoLD)

Per-transaction STF workflow (simplified)

  1. ArbOS validates format, balances, charging L2 gas and L1 posting component where applicable.
  2. Geth runs EVM execution (or Stylus WASM path for WASM contracts).
  3. ArbOS updates cross-chain state, receipts, logs.

Optimistic rollup flow

  1. Validator (or anyone) posts a rollup assertion on L1 claiming post-state after some inputs.
  2. Challenge period — others can dispute; if wrong, challenger wins via interactive game.
  3. Confirmed assertions enable L2→L1 messages and bridge withdrawals (subject to delay).

Fraud proofs & BoLD (current dispute stack)

Older “multi-day challenge” stories are outdated for Nitro’s mainnet path—docs center on BoLD (Bounded Liquidity Delay):

  • Permissionless validation — participants need not be allowlisted.
  • Interactive bisection — disagree on a large execution trace → narrow to one step.
  • One-step proof (OSP) — single step executed in WASM; L1 contract picks winner.
  • Bounded time — avoids indefinite griefing while keeping Ethereum as impartial judge.

Key insight for interviews: fraud proof is not “re-run the whole chain on L1”—it is “force agreement on one inexpensive step.”

Geth integration (what “Ethereum compatible” means)

  • Same opcode semantics for standard EVM contracts (track Arbitrum ArbOS releases for fork parity—e.g. Cancun/4844 support in ArbOS 20 / Nitro 2.3.x family).
  • Hooks implement Arbitrum-specific precompiles, gas accounting, and message parsing.
  • Storage layout discipline still matters for proxies—identical to L1 patterns.

Segment 4 — Outbox, retryable tickets, bridging, fees, interview sound bites

Parent → child (Ethereum → Arbitrum)

  • Retryable tickets — canonical L1→L2 message carrying an L2 transaction; can be retried if L2 execution fails; ~1 week redemption window; anyone can redeem; used for atomic “do something on L2 from L1” flows.
  • Canonical bridge — ETH/ERC-20 lock/mint via gateway + router architecture.
  • Direct / aliased messaging — signed EOAs or aliased contract addresses (security nuance for contracts).

Child → parent (Arbitrum → Ethereum)

  1. Contract calls ArbSys.sendTxToL1 (conceptually—see current precompile docs).
  2. Message included in a rollup assertion.
  3. After challenge period (~7 days for withdrawals in canonical bridge docs), anyone can execute on L1 with Merkle proof.

Outbox — L2→L1 execution queue on Ethereum; manual execution step reflects L1 gas limits and trust delays.

Delayed messages

Non-sequencer inbox messages are delayed relative to fast-path sequencer txs—important for fairness and censorship resistance, not for average user latency.

Fee model (dual knob)

  • Child-chain gas — pays L2 execution/storage (EIP-1559-style dynamics on L2).
  • Parent-chain data fee — pays for batch bytes on L1 (blobs or calldata); only sequencer-posted batches bear this at scale.

Related modes (do not conflate in interviews)

  • Arbitrum One — rollup with data on L1 (blobs/calldata).
  • AnyTrust / Orbit — committee DA options for chains; different trust assumption—say so explicitly.
  • Timeboost — optional ordering/MEV auction on sequencer; does not change STF semantics.

Sound like you understand compatibility + optimization

“Users get the same EVM semantics Geth enforces, but ordering and batching happen on a sequencer; we buy speed by executing offchain and buy security by posting data and assertions to Ethereum, with BoLD if someone lies about execution.”

Study checklist (Arbitrum interview)

  • Draw inbox → STF → outbox on one page.
  • Explain soft vs hard finality with real delays (feed vs L1 batch).
  • Contrast retryable ticket vs ordinary L2 tx submission.
  • Describe BoLD in three steps: assert → bisect → one-step proof.
  • Name what inherits Ethereum security (DA + disputes) vs what does not (sequencer ordering policy).

Hands-on lab (short)

(1) Read Inside Arbitrum Nitro and annotate the eight steps against your drawing. (2) Submit a cheap testnet tx via sequencer RPC; watch explorer “L1 batch” / inbox posting timing. (3) Optional: skim L1→L2 messaging and list one app flow that should use retryables.
