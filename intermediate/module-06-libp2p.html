libp2p: Transports, Identity, Security & Discovery

⏱️ Duration: 2–2.5 hours 📊 Difficulty: Intermediate

Learning Objectives

  • Understand transports (TCP, QUIC, WebSocket) and when to use which
  • Deepen PeerId and multiaddr: how identity and addressing work
  • Understand security: Noise vs TLS 1.3, encrypted channels by default
  • Learn stream multiplexing and protocol negotiation
  • Understand the DHT: Kademlia, peer routing, content routing

Transports

The transport is the layer that actually moves bytes between peers. libp2p supports multiple transports so the same application can run over different networks:

  • TCP — Reliable, widely supported; often used with a separate security layer (Noise or TLS).
  • QUIC — Built on UDP; can have built-in encryption (e.g. TLS 1.3 in QUIC). Lower latency and better for lossy networks.
  • WebSocket — Runs over HTTP; useful for browsers and environments where raw TCP is blocked.
  • WebRTC — Enables browser-to-browser and NAT traversal (e.g. hole punching).

Transports that lack native encryption (e.g. TCP, WebSocket) are secured by a libp2p security handshake (Noise or TLS) on top. QUIC may already include encryption.

PeerId and Multiaddr

PeerId is a stable identity for a peer, derived from its public key (e.g. hash of the key). It is used in the DHT, in security handshakes, and to identify who you are talking to. It does not change with network location.

Multiaddr is a self-describing address. Examples:

  • /ip4/192.168.1.1/tcp/4001 — IPv4 address, TCP, port 4001.
  • /ip6/::1/tcp/4001 — IPv6 loopback, TCP, port 4001.
  • /dns4/node.example.com/tcp/443/wss — DNS name, TCP 443, WebSocket Secure.

Peers advertise multiaddrs so others know how to dial them. A single peer can have multiple multiaddrs (e.g. IPv4 and IPv6).

Security: Noise and TLS 1.3

libp2p specifies two security handshake protocols for transports that need an explicit security layer:

  • Noise — Built on the Noise Protocol Framework. Uses the XX handshake pattern and ChaChaPoly. Protocol ID: /noise. Authenticates the peer and encrypts the channel.
  • TLS 1.3 — Standard TLS (RFC 8446). Protocol ID: /tls/1.0.0. The peer’s libp2p public key can be encoded in the certificate so PeerId is bound to the connection.

Connection establishment typically goes: (1) transport connection, (2) security handshake (Noise or TLS), (3) stream multiplexing. After that, application protocols run over multiplexed streams.

Stream Multiplexing

Multiplexing allows many logical streams (e.g. different protocols or requests) to share a single transport connection. Without it, each stream would require a new connection (more overhead, more NAT/firewall issues). With multiplexing, one connection is established and secured once, then reused for multiple streams. libp2p uses protocols like mplex or yamux for this.

Protocol Negotiation

When opening a new stream, peers must agree on which application protocol to run (e.g. /myapp/1.0.0). Multistream-select (or similar) is the mechanism: one peer proposes a list of supported protocols, the other selects one. This allows versioning and backward compatibility: new protocols can be added while old ones still work.

The DHT: Kademlia, Peer Routing, Content Routing

libp2p uses a Kademlia-style DHT (Distributed Hash Table) for:

  • Peer routing — Finding the multiaddrs (network addresses) of a peer given its PeerId. Operations like FIND_NODE: find the k closest peers to a given key (PeerId).
  • Content routing — Finding which peers have (or provide) specific content. ADD_PROVIDER / GET_PROVIDERS: advertise and look up providers for a content ID (e.g. CID).

Kademlia organizes nodes by XOR distance from a key. Each node keeps a routing table (e.g. k-buckets). Lookups take O(log n) hops in a network of n nodes. PUT_VALUE/GET_VALUE store and retrieve arbitrary signed values (e.g. peer records).

Discovery in Practice

Besides the DHT, peers can be discovered via:

  • Bootstrap list — Hardcoded or configured nodes to connect to first; from there, the DHT and gossip spread knowledge of more peers.
  • mDNS — Local network discovery.
  • Rendezvous — A designated peer or service where nodes register and find each other for a given topic.
