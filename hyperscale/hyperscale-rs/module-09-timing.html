Timing: Rounds, Heights, Timeouts & Timers

⏱️ Duration: 1–1.5 hours 📊 Difficulty: Intermediate 🎯 Hyperscale-rs Specific

Learning Objectives

  • Understand height vs round (view) and how they drive BFT progress
  • Know how timeouts trigger round advancement (view change) when no quorum is reached in time
  • See how timers (ViewChange, Cleanup) and event-driven proposals are wired in the node and BFT coordinator
  • Map timing-related config (proposal_interval, min_block_interval, view-change timeouts) and state (view, view_at_height_start) to the codebase
  • Relate timing to liveness: why timeouts and view changes are necessary for progress under delay or faults

Why timing matters

BFT consensus must make progress even when the network is slow or some validators are slow or faulty. If the leader never proposes, or proposals never reach quorum, the chain would stall. Heights and rounds structure the protocol; timeouts and timers ensure that after some bounded time the system moves to a new round (new leader) or triggers the next proposal. This module goes into detail on how hyperscale-rs does that.

For the full BFT flow (propose → vote → QC → commit), see Dynamics of Node State Machine and BFT. Here we focus on the when: rounds, heights, timeouts, and timers.

Height and round (view)

Height is the index of the block slot in the chain (block 0 = genesis, block 1, 2, …). One height = one block to be decided. Round (also called view in the code) is the attempt number for that height. If round 0 doesn’t produce a quorum in time, the protocol advances to round 1 (new proposer, new proposal) for the same height, and so on until a QC is formed.

  • Height advances only when a block at that height gets a QC and (with the two-chain rule) is committed; then the next height starts.
  • Round advances on timeout: if no QC is formed before the round timeout, validators move to the next round (view change). The proposer for round r is typically committee[(height + round) % len] (round-robin), so each round has a different leader.

In BftCoordinator (see Suggested crates / files to study), round/height state is delegated to ViewChangeController fields such as:

  • view: Round — current round (view) for HotStuff-2–style progress
  • view_at_height_start — round recorded when the current height began (backoff / logging)
  • committed_height, committed_hash — chain tip after commits

So: same height, multiple rounds = multiple attempts to get a block certified; height advances only when a block is committed (two-chain rule).

Timeouts and view change

If the proposer is slow, or the network delays messages, or not enough validators vote in time, the round may not produce a QC. To avoid waiting forever, the protocol uses a round timeout: if no QC is formed before the timeout expires, the round is abandoned and the system advances to the next round (view change).

  • Who checks the timeout? When ProtocolEvent::ViewChangeTimer fires, NodeStateMachine::on_view_change_timer (crates/node/src/state/timers.rs) calls BftCoordinator::check_round_timeout. If the leader has been idle too long, BFT returns actions that advance the local view; otherwise the node reschedules SetTimer { id: ViewChange, duration: remaining }.
  • Effect: When the round times out, the next round starts: view increments (or is updated), a new proposer is chosen, and a new proposal can be made for the same height. Validators may then vote for the new proposal; that’s a revote (allowed), not equivocation (same height, different round).

Timeouts are what give the protocol liveness: eventually, either a QC is formed (progress at this height) or the round changes (new leader, new attempt). Under partial synchrony, after GST (global stabilization time), timeouts can be chosen so that progress is guaranteed.

Timers: View-change (liveness) and Cleanup

The synchronous state machines still don’t poll a clock directly: they emit Action::SetTimer and the runner turns each deadline into a ProtocolEvent on the node.

View-change timer (round timeout)

  • When set: BftCoordinator::initialize_genesis (and recovery paths) arm TimerId::ViewChange using current_view_change_timeout().
  • When fired: The IoLoop receives ProtocolEvent::ViewChangeTimer, which routes to on_view_change_timer. That calls self.bft.check_round_timeout(topology); either the round advances or the timer is re-armed with the remaining slack.
  • Proposal attempts: There is no periodic “proposal timer” anymore. Instead, BftCoordinator::queue_ready_proposal() latches work whenever mempool/execution/provisions/BFT produce new material, and NodeStateMachine::try_event_driven_proposal gathers inputs and calls BftCoordinator::try_propose.

Cleanup timer

  • When set: Alongside the view-change timer during genesis/recovery (TimerId::Cleanup, interval from BftConfig::cleanup_interval).
  • Purpose: Housekeeping: pending-block fetch hints, sync-health checks, mempool tombstones/pending expiry sweeps (on_cleanup_timer in state/timers.rs).

SetTimer therefore covers liveness (view change) and hygiene (cleanup), while proposal timing is event-driven off new consensus inputs.

Config: proposal_interval and min_block_interval

BftConfig (and any node/simulator config that feeds it) typically includes:

  • proposal_interval — Still present in BftConfig for pacing/throttling proposals inside try_propose (wall-clock spacing between blocks when you are the leader).
  • min_block_interval — Minimum spacing between committed blocks for metrics/liveness guardrails.
  • View-change/backoff fields — Control how fast current_view_change_timeout() grows after repeated failures.

Tuning these knobs trades throughput vs. resilience: aggressive intervals stress the network; generous view-change timeouts tolerate slow links but stall longer before rotating leaders.

For field-by-field detail, read BftConfig in crates/bft/src/config.rs and the timer arms inside BftCoordinator::initialize_genesis.

Summary

  • Height = block index; round (view) = attempt for that height. Multiple rounds per height until a QC is formed; then height advances (after two-chain commit).
  • Timeouts ensure liveness: if no QC in time, round advances (view change), new proposer tries for the same height.
  • Timers: SetTimer arms ViewChange (round timeout) and Cleanup; proposals are event-driven via try_propose.
  • Config: proposal_interval, min_block_interval, and view-change/backoff parameters shape throughput vs. failover latency.
