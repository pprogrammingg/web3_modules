Crate Groups: First Contact to Cross-Shard

⏱️ Duration: 2–3 hours 📊 Difficulty: Intermediate 🎯 Hyperscale-rs Specific

Learning Objectives

By the end of this module, you will:

  • Read groups 1–6 in order (first contact, sharding, proposing, voting, execution, cross-shard)

Read the groups in order (1 → 6). Each group has a short description, code pointers, a 📖 What to read list (links to the hyperscale-rs repo), and a 10-question quiz. Pass threshold per quiz: 70%.

Group 1: First contact

Crates: production, node, mempool, types, core, messages

One-liner: How Hyperscale receives a transaction at the RPC and gets it into the right shards’ mempools.

  • production: RPC handlers (submit tx), runner that dispatches to state machine and gossips to shards.
  • node: NodeStateMachine routes ProtocolEvents (e.g. TransactionValidated, BFT votes/QCs) after IoLoop turns network/I/O into typed inputs.
  • mempool: Pool per validator, on_submit_transaction, on_transaction_gossip, involves_local_shard gating.
  • types: RoutableTransaction, declared_reads / declared_writes, topology and shard helpers.
  • core: Event, Action (e.g. BroadcastToShard, EmitTransactionStatus).
  • messages: notification types (block_header, block_vote, execution_*), TransactionGossip, OutboundMessage, batching/serialization.

Group 2: Sharding and routing

Crates: types, core, node

One-liner: Once a transaction is decomposed into NodeIds and each shard is responsible for a slice of state, these crates define who does what and where the tx is stored.

  • types: shard_for_node, Topology, consensus_shards / provisioning_shards / all_shards_for_transaction, RoutableTransaction.
  • core: Actions that are shard-aware (e.g. BroadcastToShard { shard, message }).
  • node: Uses topology.involves_local_shard(tx) and routes events to the right sub-state machines.

Group 3: Proposing and building blocks

Crates: bft, mempool, types, core

One-liner: How one validator becomes the proposer and assembles the next block from the mempool.

  • bft: HotStuff-2–style BftCoordinator; try_propose pulls ready txs/waves/provisions; emits BuildProposal; view-change timers for liveness.
  • mempool: ready_transactions(max_count, pending_commit_tx_count, pending_commit_cert_count), retries / priority / others, in-flight limits.
  • types: Block, BlockHeader, RoutableTransaction, certificate and deferral types.
  • core: Action::BuildProposal, SetTimer / CancelTimer.

Group 4: Voting and committing

Crates: bft, types, core

One-liner: How validators agree on a block and when it is finally committed (votes and QCs).

  • bft: Block verification (state root, tx root, parent QC), BlockVote, collection and VerifyAndBuildQuorumCertificate, 2-chain commit rule, BlockCommitted and chain advancement.
  • types: QuorumCertificate, BlockVote, signing message format, BlockHeader.
  • core: Action::VerifyQcSignature, VerifyAndBuildQuorumCertificate, PersistAndBroadcastVote, Event::BlockCommitted.

Group 5: Execution after commit

Crates: execution, engine, node, types, core

One-liner: Once a block is committed, who runs the transactions and how single-shard vs cross-shard paths diverge.

  • execution: on_block_committed, split single-shard vs cross-shard, single-shard execution and StateVoteBlock, cross-shard registration and provision-based (five-phase) coordination.
  • engine: Radix Engine integration, execute_single_shard, execution output and state writes.
  • node: Dispatches BlockCommitted to execution, handles TransactionExecuted and certificate feedback to BFT/mempool.
  • types: StateVoteBlock, StateCertificate, TransactionCertificate, TransactionDecision.
  • core: ExecuteTransactions, BroadcastStateVote, AggregateStateCertificate, EmitCommittedBlock.

Group 6: Cross-shard (provisions and livelock)

Crates: provisions, execution, types, core — plus simulator for post-run livelock analysis (not a consensus crate).

One-liner: How state moves between shards and how dependency cycles are broken: ConflictDetector in execution (conflict.rs), provisions + waves, status via core — simulator’s livelock.rs only analyzes stuck txs after a run.

  • provisions: ProvisionCoordinator, receiving and buffering StateProvision, quorum per source shard, VerifyAndAggregateProvisions, CommitmentProof.
  • execution: Cross-shard provision-based phases, ConflictDetector for bidirectional overlaps, wave abort integration; coordinates with provisioning (provisioning.rs).
  • simulator (tooling): LivelockAnalyzer in simulator/src/livelock.rs, --analyze-livelocks — diagnostics, not the live protocol path.
  • types: StateProvision, state / wave certificates, routing manifests — align names with current crates/types exports.
  • core: BroadcastStateProvision, BroadcastStateCertificate, VerifyAndAggregateProvisions, transaction status actions (EmitTransactionStatus includes deferred/retry semantics).

OSS contribution: simulator trait & deduplication (#23)

You’ve mapped the crate groups—nice work. This contribution path rewards patience over heroics: a thin trait and one migrated callsite beats a thousand-line refactor nobody can review. Work alongside maintainers in the open; drafts exist so you can learn in public without pretending perfection.

After the quizzes, use #23 — Common trait for parallel and deterministic sims as your umbrella issue.

Step 1 — Map the fork in the road

Sketch (on paper or in the issue) where parallel vs deterministic runners diverge: simulator, simulation, worker pools, time injection—whatever the code actually does.

Expected result: You can name two concrete files or types where behavior splits; no hand-wavy “the sim does stuff.”

Step 2 — Shape the trait

Propose a trait both harnesses could implement: boundaries for event injection, simulated time, and the network adapter—only what both sides truly share.

Expected result: A short trait sketch in the issue (or a gist) that reviewers can react to before you engrave it in stone.

Step 3 — Draft PR, tiny blast radius

Open a draft PR with the trait plus one migrated call site so people see direction without a big-bang diff.

Expected result: CI runs; discussion focuses on design, not fatigue; you stay energized for the next slice.
The rewarding bit: When two harnesses share a language, debugging and new tests get easier for everyone—including you on your next PR.
