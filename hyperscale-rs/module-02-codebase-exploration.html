Exploring the Hyperscale-rs Codebase

⏱️ Duration: 1.5-2 hours 📊 Difficulty: Intermediate 🎯 Hyperscale-rs Specific

Learning Objectives

  • Navigate the codebase effectively
  • Understand core types and traits
  • Learn how NodeStateMachine works
  • Read and understand Rust code patterns used in the project

1. Where to Start

After cloning and building hyperscale-rs, start from the core and node crates: re-exports and the StateMachine trait, then NodeStateMachine composition, then shared types. Exact paths are in Suggested crates / files to study below.

Use cargo doc --open to generate and open API docs for the workspace; search for types and traits by name.

2. Core Types and Traits

StateMachine trait

Everything that participates in consensus implements the StateMachine trait. In the core crate’s traits module you’ll see:

fn handle(&mut self, event: Event) -> Vec<Action>;
fn set_time(&mut self, now: Duration);
fn now(&self) -> Duration;

Events flow in; actions flow out. Time is explicit for determinism.

🤔 Think about it: Why pass now in from the outside instead of calling something like SystemTime::now() inside the state machine? What would break in tests or replay if the code read the real clock?

Event and Action enums

In the core crate’s event and action modules you’ll see the full set of events (e.g. BlockHeaderReceived, ProposalTimer, QuorumCertificateFormed) and actions (e.g. BroadcastToShard, SetTimer, CommitBlock). Browsing these gives you a map of what the system can react to and what it can do.

3. NodeStateMachine and Composition

The NodeStateMachine in the node crate composes several sub–state machines (BFT, execution, mempool, etc.). When an event arrives:

  1. The node routes the event to the appropriate sub–state machine(s)
  2. Each returns a list of actions
  3. Actions are collected and executed by the runner (simulation or production)

Open the node state machine’s handle implementation (see study table). Trace one event type (e.g. Event::BlockHeaderReceived) to see which sub–machines receive it and what actions they return.

🤔 Check your understanding: If the node received two events at the same logical time, why would the *order* in which they're processed matter? (Think about proposer logic and votes.)

4. Rust Patterns You'll See

  • Enums for events/actions – sum types with payloads; match on them to handle each case
  • No async in core – consensus logic is synchronous; async lives in runners and I/O
  • Explicit time – set_time / now so that tests and production use the same deterministic time
  • Pure functions – handle takes state + event and returns actions; no hidden I/O

5. Practical Exploration Tasks

Tasks

  1. Run cargo doc --open and open the core and node crates. Find the StateMachine trait and the NodeStateMachine struct.
  2. In the node state machine source (study table), find where Event::ProposalTimer or Event::BlockHeaderReceived is handled. Note which sub–state machine(s) receive it.
  3. In core’s event and action modules (study table), list three event variants and one field or associated type for each, and the same for three action variants.
  4. In your learning journal, write one paragraph: “How does the NodeStateMachine decide where to send an event?” (Use the code to support your answer.)

Success Criteria

  • ✅ You can open the docs and find core types
  • ✅ You can trace one event through the node’s handle path
  • ✅ You can name several Event and Action variants and their role
