Cryptography in Hyperscale-rs

⏱️ Duration: 1.5–2 hours 📊 Difficulty: Intermediate 🎯 Hyperscale-rs Specific

Learning Objectives

  • Understand why BLS is used for quorum and state certificates
  • Compare signature schemes used in blockchain (table)
  • Understand why libp2p keypair is derived from BLS public key
  • See where BLS and Ed25519 appear in the repo
  • Understand QC and StateCertificate aggregation

Why BLS is chosen

The repo doesn’t spell out “we chose BLS because …” in one place, but the usage makes the reason clear: aggregatable signatures for consensus.

  • QuorumCertificate (e.g. in crates/types/src/quorum_certificate.rs) holds a single Bls12381G2Signature that is an aggregated signature from 2f+1 validators for the same message (block hash, etc.).
  • StateCertificate (in crates/types/src/state.rs) also uses an aggregated BLS signature from many StateVoteBlocks.

Flow:

  1. Many validators sign the same message (e.g. block hash).
  2. Their BLS signatures are combined into one aggregated signature.
  3. You store and verify one signature and one aggregated public key, instead of N Ed25519 signatures.

That gives:

  • Constant size and cost — QC size and verification cost don’t grow with committee size.
  • Compact quorum certificates — One 96-byte BLS G2 signature instead of N×64-byte Ed25519 signatures.
  • Same-message aggregation — Matches the “everyone signs this block hash” pattern (as in Ethereum 2, Chia, etc.).

In the types module, the split is summarized as: Ed25519 for “fast signing” and BLS (BLS12-381 G1 pubkey / G2 sig) for “aggregatable signatures”. So BLS is chosen for quorum/state certificates; the keygen binary generates the validator BLS key (from a 32-byte hex seed or at random) used in that aggregation scheme.

Signature schemes: comparison

Different schemes are used for different roles. The table below summarizes pros and cons so you can see why BLS fits consensus certificates and why Ed25519 is used where aggregation isn’t needed.

Scheme Typical use Pros Cons
Ed25519 Tx signing, fast one-off signatures Fast signing/verify; small keys & sigs; no pairing No native aggregation; N sigs = N times size/cost
BLS (BLS12-381) QC, StateCertificate, attestations Same-message aggregation; constant-size QC; one verify for 2f+1 signers Slower than Ed25519; pairing-based; need same message for aggregation
ECDSA (e.g. secp256k1) Bitcoin, Ethereum (legacy) tx signing Widely used; hardware support; no aggregation needed for single-sig txs No aggregation; larger sigs than Ed25519; malleability concerns

In Hyperscale-rs: use the table to justify “BLS for QC/StateCertificate, Ed25519 for fast signing” where the design is explained (e.g. in docs or a future general Cryptography module).

Why we derive libp2p keypair from BLS public key

One identity, two roles. Consensus identity is the BLS public key (votes, QCs, validator set). libp2p needs an Ed25519-style keypair for TLS and PeerId, not BLS. Instead of two unrelated keys, the libp2p keypair is derived from the BLS public key — one logical identity, with the p2p key as a deterministic view for transport.

Derivation uses the public key only (no secret): hash the BLS public key bytes with a domain separator, use the hash as seed for an Ed25519 keypair (see adapter.rs). In the code this separator is LIBP2P_IDENTITY_DOMAIN = b"hyperscale-libp2p-identity-v1:": it is prepended (or otherwise combined) with the pubkey before hashing so that the derived key is bound to this specific use. That way the same BLS pubkey hashed with a different domain would yield a different PeerId, avoiding collisions and tying the derivation to “Hyperscale libp2p identity” only. So anyone who knows a validator’s BLS public key can compute that validator’s PeerId (e.g. compute_peer_id_for_validator(public_key)). The topology already has every validator’s BLS public key, so at startup the runner can fill validator_peers and peer_validators without out-of-band PeerId exchange.

In production: the validator binary builds the libp2p identity from the same BLS key it uses for consensus (derive_libp2p_keypair(&signing_keypair.public_key()) in validator.rs). The runner registers validator_id → peer_id; the network layer uses that mapping for peer validation (only known validators in peer_validators).

Why not BLS for libp2p? libp2p’s identity and TLS are built for key types like Ed25519. So: BLS = consensus identity; Ed25519 = derived only for transport. The stack works without changing libp2p, and every validator has a deterministic, derivable PeerId from their public key.

Where to look in the repo

  • crates/types/src/quorum_certificate.rs — QC type and BLS aggregated signature.
  • crates/types/src/state.rs — StateCertificate and aggregated BLS.
  • crates/production/src/adapter.rs — derive_libp2p_keypair, LIBP2P_IDENTITY_DOMAIN (domain separator), PeerId from BLS pubkey.
  • crates/production/src/validator.rs — validator builds p2p identity from BLS key.
  • Keygen / crypto crate — validator BLS key generation (32-byte hex seed or random).
