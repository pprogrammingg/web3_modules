Your First Contribution: Documentation & Tests

⏱️ Practical track: ~1 hour · With deep reading: 2–4 hours 📊 Difficulty: Level 2 🎯 Hyperscale-rs Specific

You belong here. Every maintainer’s first merge started with a small, careful change—documentation, a test, a typo fix. This module walks beside you: each step says what to do and what “good” looks like when you’re done. There is no prize for rushing; the win is shipping something real and feeling confident doing it again.

Learning Objectives

By the end, you will have:

  • Contrast simulation vs production E2E harnesses and where those tests live in the repo
  • Produce two short numbered traces (sim + prod) that name concrete symbols and setup phases—not only high-level summaries
  • Contribute one new simulation E2E and one production E2E extension that fit existing patterns and CI
  • Ship both through a forked PR using Section 3 — Submitting Your PR and respond to review

1. Docs, tests, and end-to-end (this module’s focus)

Documentation and small unit tests are still the lowest-risk way to learn layout, review culture, and Rust conventions—but this module’s assignment centers on integration / E2E tests: same core state machine as production, exercised through the simulation harness (fast, deterministic) and the production harness (real threads, network, storage). Line-by-line tracing of canonical tests lives in End-to-end tests: tracing code to flow; read or skim it alongside Segments 1–2 below if you have not already.

2. Practical Assignment

Assignment: four segments (~1 hour)

What “trace” means here: Segments 1–2 ask for numbered traces in your journal—steps tied to symbols and blocks you read in the source. That deliverable is prose, not extra prints in the repo. If you add temporary instrumentation while exploring, use tracing::debug! (see use tracing::debug) and run with RUST_LOG=debug, consistent with module 1 — Trace flows end-to-end and module 11—not println!, which bypasses log filtering and is easy to leave behind by mistake.

Work in order. Each segment is sized for about 10–25 minutes—one small win per sitting: a trace you can narrate, a command whose story you understand, then two tight contributions. Checkboxes save in this browser (resume anytime). Deep reference material lives in module 11 — End-to-end tests; keep it open in another tab.

Segment 1 — Simulation: trace one canonical E2E

⏱️ ~10–25 min

By the end: You ran test_e2e_single_shard_transaction (or another substantial test in the same file) and can name where setup ends and where the harness drives ShardLoop::run_step toward commit.

Where to open: crates/simulation/tests/e2e_tests.rs. Jump to fn test_e2e_single_shard_transaction (editor search). Skim the module docs (//!) first—they frame what “good” means for this harness.

Anchor pattern — simulation tests usually spin a runner, advance simulated time, and inject work; your trace should follow that spine:

// Illustrative shape — follow the real test body in e2e_tests.rs
#[test]
fn test_e2e_single_shard_transaction() {
    // 1. Build SimulationRunner + SimulatedNetwork + storage fixtures
    // 2. Genesis / shard setup to match the scenario
    // 3. Submit tx / drive inputs (same APIs sibling tests use)
    // 4. runner.run_until(...) or stepping loop — each tick hits ShardLoop::run_step
    // 5. flush / settle — simulation flushes batches aggressively each step
    // 6. assert committed state, receipts, or cross-check invariants
}

Numbered trace (do this in your journal): Write 6–10 steps. For each step, note which symbol or block you are in (example hints below—not line numbers, because they drift between commits).

  1. Fixture arrives: you see SimulationRunner / builder calls and network/storage wiring.
  2. Clock domain: simulated Duration advances—no wall clock.
  3. Inputs: transactions or control messages enter the same paths production uses once inside step.
  4. Step boundary: each iteration ultimately calls into ShardLoop::run_step (find the call site in the runner loop).
  5. After step: simulation-style flushing drains batches immediately—tie this to what you read in module 11.
  6. Proof: assertions on balances, logs, or committed blocks—whatever this test encodes as “done.”

Run it:

RUST_LOG=debug cargo test -p hyperscale-simulation --test e2e_tests test_e2e_single_shard_transaction -- --nocapture

What you should see: Cargo compiling on first run (coffee moment), then test test_e2e_single_shard_transaction ... ok. RUST_LOG=debug surfaces tracing::debug! lines when you add temporary instrumentation; omit the prefix if you want a quieter run.

Segment 2 — Production: same exercise, heavier harness

⏱️ ~10–25 min

By the end: You can contrast this test’s startup story with Segment 1: pinned loop, wall clock, real network stack, RocksDB—without losing the thread that “inside step, the story matches.”

Where to open: crates/production/tests/e2e_tests.rs. Jump to fn test_production_runner_with_network.

Anchor pattern — production tests pay once for process-realistic wiring; your trace should capture genesis → runner → pinned thread → shutdown:

// Illustrative shape — follow the real test in production e2e_tests.rs
#[test]
fn test_production_runner_with_network() {
    // 1. ProductionRunner::build / temp dirs / RocksDB open
    // 2. Network + storage adapters — Libp2p path, not SimulatedNetwork
    // 3. Genesis and shard bootstrap (may run on async side before pin)
    // 4. spawn_shard_loop → run_shard_loop drives ShardLoop::run_step on std::thread
    // 5. Submit work; wall-clock batch flushing (flush_expired_batches)
    // 6. Graceful shutdown + asserts on live behavior
}

Can you run it? Yes, on a normal developer machine that already builds the workspace: it is a standard cargo test integration test. Expect a slower compile and test runtime than simulation; native deps (for example RocksDB) must build. Use a single test thread while iterating:

RUST_LOG=debug cargo test -p hyperscale-production --test e2e_tests test_production_runner_with_network -- --test-threads=1 --nocapture

If the binary fails to link on your machine, finish this segment using module 11’s walkthrough + reading the test source, write your numbered trace anyway, and note the blocker in your journal—maintainers see this occasionally; CI still proves the path.

Numbered trace: Again 6–10 steps, plain English. Explicitly call out one moment where Tokio/async hands work to the pinned shard-loop (channels), per module 11.

Segment 3 — Add one new simulation E2E

⏱️ ~10–25 min

By the end: You have a new #[test] next to the ones you studied—small, reviewable, green locally.

Add it to crates/simulation/tests/e2e_tests.rs (or the workspace’s established split if tests moved). Reuse SimulationRunner patterns from neighbors in the same file. Target a real gap: behavior not yet asserted, or deeper coverage of a flow already supported by test APIs—grep siblings for transaction builders; do not invent unsupported payloads.

Scope questions: tie to issue #18 or ask early on a draft PR.

RUST_LOG=debug cargo test -p hyperscale-simulation --test e2e_tests <your_test_name> -- --nocapture

What you should see: Your test name in the summary line ending in ok; failures print panics/asserts—fix by aligning with existing fixtures before widening ambition.

Segment 4 — Extend production E2E + open the PR

⏱️ ~10–25 min

By the end: One focused production change (assertion, invariant, or short scenario on existing setup) is passing under the prod command, and you have (or are ready to open) a PR from your fork.

Prefer a surgical edit in crates/production/tests/e2e_tests.rs—extend test_production_runner_with_network’s setup rather than spawning new multi-node complexity unless a maintainer agreed.

RUST_LOG=debug cargo test -p hyperscale-production --test e2e_tests <your_test_name> -- --test-threads=1 --nocapture

What you should see: Same as Segment 2 style output—your test name listed with ok. Keep using --test-threads=1 until you are confident the test is deterministic.

Then follow Section 3 — Submitting Your PR below: fork workflow, PR body with both cargo test lines you ran, link to issue when helpful.

Why these four? Segments 1–2 build reading muscle and a personal map of the repo; segments 3–4 turn that map into commits CI can replay. A fifth “reflection” step is optional—fold a sentence into your PR description instead.

When stuck: Ship simulation first; mirror to production only after green. Follow existing builders; keep tests deterministic.

Success criteria

  • ✅ Two numbered traces (sim + prod) with code anchors—not only prose summary
  • ✅ One new simulation E2E and one production E2E extension, both green with the commands above (or documented env blocker for prod)
  • ✅ PR from fork includes verification commands and issue link when scope was non-obvious
The sweet part: You walked the same paths CI walks—fast simulation for iteration, production harness for reality—and left evidence others can replay.

3. Submitting Your PR

After you’ve carried out the practical assignment above, use this checklist to open and land your contribution (include the exact RUST_LOG=debug cargo test -p hyperscale-simulation … and RUST_LOG=debug cargo test -p hyperscale-production … commands you ran).

Step 1 — Fork upstream, then branch on your copy

Upstream is the canonical project repo (e.g. hyperscale-rs in the org). As an outside contributor you almost never get permission to push branches straight there—you wouldn’t be able to anyway without maintainer write access.

What to do instead: On GitHub, fork the upstream repo into your account. Clone your fork (or add it as origin), create a branch with a clear name such as docs/improve-core-traits or tests/add-event-coverage, commit there, and push to your fork. Then open a pull request from your fork’s branch into upstream. One logical change per PR.

Maintainers with direct push access may branch on the upstream repo; everyone else should stick to fork → PR.

Expected result: Your pushes succeed because they target a repo you control; reviewers see a normal PR from fork to upstream; your branch name still tells a story at a glance.

Step 2 — Describe and link

In the PR body: reference the issue (Fixes #42 / Closes #42 when appropriate), say what you changed, why, and how you verified it—for example: “Added unit tests for handle(ProposalTimer) when node is proposer; ran cargo test, green.”

Expected result: A reviewer can approve without guessing your intent; you look thorough, not flashy.

Step 3 — Commit messages that match the house style

Follow the repo’s CONTRIBUTING.md for commit and PR conventions. Common shape: short imperative subject (<72 chars), optional body—e.g. docs(core): clarify StateMachine trait contract or test(bft): add ProposalTimer proposer path.

Expected result: CI and maintainers recognize the format; your commits read like part of the project history.

Step 4 — Run checks and respond with grace

Run cargo test, cargo doc, and any other checks the repo documents. When feedback arrives, treat it as collaboration: ask if something is unclear, push fixes in small commits or as requested.

Expected result: CI passes; conversation stays constructive; you finish feeling respected even if the last tweak is tiny.
