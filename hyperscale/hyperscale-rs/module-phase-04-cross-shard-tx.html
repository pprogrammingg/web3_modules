0 — Prerequisite mental model

  • No global mempool. Each shard committee has its own mempool and BFT instance. A cross-shard tx is the same bytes replicated to every shard it touches; admission is local.
  • No 2PC coordinator. Atomicity comes from provision-based composition: remote shards commit read/write snapshots in their blocks, target shards verify merkle proofs, then waves execute together. There is no single “transaction manager” process.
  • Trust remote state via headers + QC. Target shards learn source-shard commits through CommittedBlockHeaderGossip (header + certifying QC + sender BLS attestation). Provisions reference committed state_root and inclusion proofs — not honor-system RPC.
  • Time is still BFT time. Provision deadlines and execution timeouts use weighted timestamps from commits, aligned with Phase 3.

1 — Step 6: gossip to every touched shard

Proof snippets are trimmed from the hyperscale-rs tree.

When a node accepts a user transaction, it computes the shard set from declared_reads ∪ declared_writes (hash-modulo routing of NodeId). For each shard in that set, it enqueues gossip — same validation path as Phase 1, repeated per destination committee.

A transaction is “cross-shard” when that set has more than one shard id. Single-shard txs never leave the local committee for admission (though they still use the same code path with a set of size one).

In-repo proof — per-shard enqueue after submit

// crates/node/src/shard_loop/step/tx_validation.rs — handle_submit_transaction
let shards: BTreeSet<ShardGroupId> = tx.declared_reads().iter()
    .chain(tx.declared_writes().iter())
    .map(|node_id| shard_for_node(node_id, num_shards)).collect();
for shard in shards {
    self.enqueue_tx_for_gossip(shard, Arc::clone(&tx));
}
// crates/types/src/topology/snapshot.rs
pub fn is_cross_shard_transaction(&self, tx: &RoutableTransaction) -> bool {
    self.all_shards_for_transaction(tx).len() > 1
}

2 — Steps 7–11: each shard runs its own BFT

After gossip, every involved shard runs Phase 1 admission and Phase 2 propose → vote → QC → commit independently. Heights, rounds, and leaders are per shard. A slow shard does not stop another shard’s chain from advancing, but a cross-shard tx cannot execute on shard B until B has the provision data it needs from shard A’s commits.

There is no cross-shard block proposal — only cross-shard messages (gossip, provisions, execution certificates, committed headers).

3 — Committed headers: how target shards learn source commits

When a block commits on shard S, the block proposer gossips CommittedBlockHeader { header, qc } globally (BroadcastCommittedBlockHeader). Receiving shards verify the sender’s BLS attestation over committed_block_header_message(shard, height, block_hash), then ingest the header through the remote-headers pipeline.

That header carries the state_root and provision_root remote validators need to check inclusion proofs. This is separate from block-vote BLS (Phase 2) and from user-tx Ed25519 (Phase 1).

Gotcha — proposer vs every validator: Every validator commits locally on two-chain; only the proposer is required to broadcast the committed header for cross-shard liveness. Slow or Byzantine proposers trigger fallback fetch paths in RemoteHeaderCoordinator.

In-repo proof — proposer-only broadcast after commit

// crates/shard/src/coordinator.rs — commit_one_buffered_block
if block.header().proposer() == topology_snapshot.local_validator_id() {
    let committed_header = CommittedBlockHeader::new(block.header().clone(), qc);
    actions.push(Action::BroadcastCommittedBlockHeader { committed_header });
}

4 — Provisions: ProvisionCoordinator and quorum per shard

After a source block commits, validators emit state provisions — payloads that prove specific substate for cross-shard readers. The ProvisionCoordinator tracks, per requirement, which remote shards have delivered verified provision data before execution may proceed.

Provisions can also be **included in block bodies** via provision_root so all validators on a shard share the same bytes (Byzantine-safe replication). The coordinator prunes committed hashes from the proposer queue, drops past-deadline garbage (prevents endless view-change loops on unverifiable proposals), and schedules fallback fetches when headers are late.

In-repo proof — commit fan-out includes provisions

// crates/node/src/state/shard.rs — on_block_committed
actions.extend(
    self.provisions.on_block_committed(self.topology.snapshot(), certified),
);
self.outbound_provisions.on_block_committed(certified.qc().weighted_timestamp());

5 — Five-phase execution protocol (no 2PC)

The execution crate documents the cross-shard lifecycle explicitly. Phases 1–2 are provisioning + conflict detection; Phase 3 is wave-atomic Radix dispatch; Phases 4–5 are execution votes and cross-shard finalization. All of it is driven by events and timeouts on each validator — not a centralized coordinator.

  1. State provisioning — proposer broadcasts proofs; targets verify against committed headers.
  2. Conflict detection — ConflictDetector aborts overlapping txs deterministically.
  3. Wave-atomic execution — ExecuteCrossShardTransactions when every required provision is present.
  4. Vote aggregation — wave leader builds EC at 2f+1 on receipt root.
  5. Finalization — collect remote shard proofs; finalize_wave → FinalizedWave.

6 — Hooking back to Phase 3 waves

On commit, cross-shard txs register record_required shard sets and may abort early if conflicts demand it. WaveState::absorb_ready_provisions transitions the wave when the last remote shard reports in. Until then, the wave stays idle and WAVE_TIMEOUT may abort the whole wave — freeing the cluster from waiting forever on a missing remote.

When provisions land mid-block, later events call back into execution to dispatch and scan for votes — the wave machine is incremental, not “commit only once.”

7 — Manifest order vs protocol order (not in Phase 3)

The Radix Engine runs manifest instructions sequentially (step 2 depends on step 1). Hyperscale consensus does not preserve that instruction order. Instruction analysis walks the manifest and collects declared_reads and declared_writes as sets of NodeIds; shard sets come from shard_for_node and are stored in BTreeSet order — coordination proceeds by ShardGroupId, not “account shard first, then LP shard.”

Prerequisites between shards are enforced by provisions, not by manifest line numbers: shard B cannot finish until it has quorum proofs from every other shard in required_shards.

9 — Design Q&A (curated)

Is there a global transaction coordinator?

No. Composition is provision-based: each shard’s BFT commits local evidence; targets verify proofs; execution waves finalize with multi-shard EC agreement. That is the “atomic composability” story without a classic 2PC leader.

Why gossip the tx to shards that only read remote state?

Because admission, mempool ordering, and BFT inclusion are per-shard. A read on shard B still requires B’s validators to see and validate the transaction bytes under B’s rules before it can execute there after provisions arrive.

What if the source shard committed but the header gossip is slow?

ProvisionCoordinator records expectations and, after timeout, emits fallback Fetch(RemoteProvisions) (and related abandon paths on cleanup). Execution can also flush expected provisions under sync urgency. The system favors liveness with deterministic timeouts over waiting on one gossip path.

How is this different from Phase 2 committed-header crypto?

Phase 2 explained three BLS paths (votes, parent QC, committed-header relay). Phase 4 uses the third so shard B can trust shard A’s state_root before accepting provisions — then provisions unlock Phase 3 execution on B.

10 — Design debate: provision-based vs 2PC

Provision-based

No single point of failure; each shard’s QC evidence is locally verifiable; aligns with sharded BFT replication.

Cost

More moving parts (headers, proofs, timeouts, conflict aborts). Debugging “stuck cross-shard tx” needs the full pipeline diagram.

Mitigation: Follow expectations in ProvisionCoordinator + execution expected_certs metrics; sim livelock analysis tools.

11 — Phase 4 checklist

  • Submit path gossips to all shards in declared_reads ∪ declared_writes
  • Each shard: Phase 1 admit + Phase 2 commit independently
  • Proposer broadcasts CommittedBlockHeader; receivers verify sender BLS + QC
  • ProvisionCoordinator tracks quorum / timeouts / queue prune on commit
  • Execution waves dispatch only when provisions + conflict rules satisfied
