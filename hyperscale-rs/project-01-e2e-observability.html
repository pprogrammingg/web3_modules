Hands-on project 1: E2E observability & invariants

⏱️ Duration: 1.5–2 hours 📊 Difficulty: Intermediate Hands-on project Hyperscale-rs

Why this project

You already traced test_e2e_single_shard_transaction on paper in End-to-End Tests. Here you change the real test in the repo: add visible trace points, add a committed-height check across validators (and learn when strict equality matches what a single run_until snapshot can show), and relate SimulationStats to how much simulated time you allow. Everything runs under cargo test with --nocapture so you see effects immediately—no extra harness.

Prerequisites

  • Local clone of hyperscale-rs, cargo and stable Rust working.
  • Comfort reading Rust integration tests and panics.
  • Skim module 11 so SimulationRunner, run_until, and SubmitTransaction are not new.

Files you will touch

  • crates/simulation/tests/e2e_tests.rs — function test_e2e_single_shard_transaction (starts near the top of the single-shard tests section).
  • Optional: crates/simulation/src/runner.rs — SimulationStats (Debug) if you want to read field meanings in source.

Step 1 — Baseline run

From the hyperscale-rs repository root:

cargo test -p hyperscale-simulation test_e2e_single_shard_transaction -- --nocapture

Confirm the test passes. Scroll the output once: note genesis message, transaction hash, “initial consensus”, mempool status, and the final pass banner. You will align your own prints with this rhythm.

Step 2 — Trace markers

In test_e2e_single_shard_transaction, add two clear markers (either println! or tracing::info!):

  1. Immediately after runner.initialize_genesis(); — e.g. println!("[lab] genesis done");
  2. Immediately after the runner.schedule_initial_event(... SubmitTransaction ...) call — e.g. println!("[lab] submit scheduled on node 0");

Re-run the same cargo test command. Verify [lab] genesis done appears before [lab] submit scheduled and before the long simulation output. If you use tracing, set RUST_LOG=info for that crate as needed; println! is enough for this lab.

Step 3 — Committed heights across validators

With full connectivity and shared simulated time, validators eventually agree on the same committed tip—but a single call to run_until gives you a snapshot at a discrete event boundary, not a proof that every node has applied every message by that instant.

Important caveat

Why you might see [2, 2, 2, 1] (or similar) right after run_until(2s)

Commits propagate through the simulated network with nonzero latency. When simulated time hits your cutoff, the message that would advance the last validator’s committed_height() may still be scheduled for delivery after that cutoff. The runner stops when the next event time would exceed end_time (see run_until in crates/simulation/src/runner.rs)—so one node can legitimately be one block behind at that moment. That usually means eventual sync, not a fork or a broken chain.

Takeaway: requiring heights.iter().all(|h| h == heights[0]) immediately after a fixed Duration::from_secs(2) is stricter than the harness guarantees. If your assertion panics with one node one height lower, the lesson is real; adjust the lab so the test stays meaningful and green.

Ways to align the exercise with the simulator

  • Eventual equality: After the first run_until(2s), loop with small extra run_until steps (or one longer slice) until all heights match, with a safety cap on iterations so the test cannot hang.
  • Weaker invariant: Assert that the highest and lowest committed_height in your vector differ by at most one, and print the vector—then explain in notes why a spread of one can happen at a hard time stop.
  • Strict equality after more time: Keep the strict all(|h| h == heights[0]) only once you have enough simulated time (or the spin loop above) that all four nodes have caught up.

After the first

runner.run_until(Duration::from_secs(2));

insert code that collects committed_height() for node indices 0..4 (this test uses four validators on one shard). Use runner.node(i).expect("node").bft().committed_height().

Requirements:

  • Always print the Vec of heights (e.g. println!("[lab] committed heights … {:?}", heights)) so you can see alignment or lag at a glance.
  • Use an assertion strategy you can justify: either strict equality after a spin/longer run, or a documented weaker check, per the caveat above. If you use strict equality right at 2s only, expect occasional failure unless your network timing always delivers in time.
  • Whatever you assert, the panic or failure path should still print the full height vector so a mismatch is obvious.

Example shape (adapt names, message, and assertion to match the option you chose):

let heights: Vec<u64> = (0..4u32)
    .map(|i| runner.node(i).expect("node").bft().committed_height())
    .collect();
// Strict snapshot (may fail under latency + fixed cutoff—see caveat above):
assert!(
    heights.iter().all(|&h| h == heights[0]),
    "validators not aligned on committed height: {:?}",
    heights
);
println!("[lab] committed heights after 2s sim time: {:?}", heights);

If your test sometimes returns early (transaction already executed), your check still runs on the path that reaches it—if you hit an early return before this line, move the height logic to a small helper and call it from each exit path, or document which path you tested; for grading yourself, the main path after the first run_until is enough.

Step 4 — SimulationStats vs time

Still after that first run_until(2s) (and ideally after your height assertion), print runner.stats():

println!("[lab] stats after 2s: {:?}", runner.stats());

Run again and note events_processed, messages_sent, and timers_set.

Now change only that first run_until duration to Duration::from_millis(15) (very tight). Re-run with --nocapture.

  • Are committed heights still equal across nodes (or within your Step 3 tolerance)? (With only 15ms they often still match but stay at 0.)
  • How do the stats compare to the 2-second run?
  • Does the test still pass end-to-end, or does it fail later because the transaction never finishes?

Restore Duration::from_secs(2) before you finish so you do not leave the suite flaky for others.

Step 5 — Stretch (optional)

  • Repeat the height check a second time after the polling loop completes (or before final assertions), and confirm heights still satisfy your Step 3 rule (strict match or bounded spread) at the end of the test.
  • Or: temporarily remove / comment out the schedule_initial_event for SubmitTransaction and observe how failure mode differs (mempool never sees the tx vs sync invariant).

What “done” looks like

  • Markers in output prove you control where genesis and submit appear relative to simulation progress.
  • You can describe committed heights across the four nodes after a run_until slice, and explain why strict equality at an arbitrary time limit can fail under simulated gossip latency even when validators are not disagreeing on the canonical chain.
  • You can explain in one paragraph why starving simulated time lowers events_processed and can prevent the transaction from reaching executed state even when heights look aligned.
