Important background & concepts

Segment 1 — Mental model (most important)

Ethereum execution is a deterministic replicated state machine: every honest node applies the same well-ordered transactions and arrives at the same post-state commitment (state root), modulo specs and sync edge cases.

Core pipeline (conceptual): transaction → mempool / builder economics → inclusion in a block → EVM execution + state transition → commitments (Patricia trie roots, receipts, logs bloom in the execution block header tradition).

Key insight: the EVM is not free-form “backend compute.” It is consensus-constrained computation: every step must be replicated, metered (gas), and committed under tight rules so all nodes agree.

Accounts

  • EOA — controlled by a key; has nonce, balance; no contract bytecode in the account’s code field.
  • Contract account — holds bytecode and persistent storage; behavior driven by code.
  • Per-account fields (Yellow Paper / EL view): balance, nonce, code hash, storage root — together they fix what the world trie stores for that address.

Segment 2 — Storage, memory, calldata, transient storage, gas, atomicity

Storage (persistent)

  • Persistent across transactions; Merkle-committed via per-contract storage trie (conceptually keyed under the account in the state trie).
  • Expensive — SLOAD / SSTORE dominate many real workloads; pricing includes warm/cold distinctions and (where applicable) gas refunds—still treat storage as scarce.

Memory (frame-local)

  • Ephemeral — expanded by words, zero-initialized on first touch; not shared across calls as a single flat buffer. Each activation has its own memory; when the frame returns, that memory is gone.
  • Cheap relative to storage writes, but not free — quadratic pricing on expansion protects against abuse.

Calldata

  • The read-only tx input bytes to a call; cheap per-byte compared to storage; central to ABI decoding.

Transient storage (EIP-1153)

  • Transaction-scoped scratch keyed storage (not committed to the trie). Useful for reentrancy locks, temporary accounting, router patterns — cheaper than persistent storage when you only need values for the rest of the tx.

Gas model

  • Gas bounds work and bounds state growth; it is the practical answer to undecidable “will this halt?” — you pay until you run out.
  • Modern EL philosophy: state growth and I/O to committed state often matter more than raw arithmetic — but profile real contracts; hotspots vary.

Execution model

  • Per-tx execution is single-threaded, deterministic, and atomic: on revert, state changes from that transaction are rolled back per the execution spec (always cite the fork you mean if you discuss edge cases).

Segment 3 — Calls, upgradeability, ABI, logs, reentrancy

CALL vs DELEGATECALL (critical)

  • CALL — run target’s code in the target’s account context (code, storage, balance rules per opcode semantics).
  • DELEGATECALL — run callee bytecode but keep the caller’s storage / address context for storage ops — the “golden sentence”: delegatecall separates code identity from storage identity.
  • Powers proxies, UUPS / transparent patterns, diamonds (EIP-2535), plugins — all demand storage layout discipline.

Upgradeability patterns (interview framing)

  • Transparent proxy — admin vs user routing; implementation holds logic; proxy holds state.
  • UUPS — upgrade logic often lives in the implementation with careful slot choices.
  • Diamond — selector → facet routing; still delegatecall-shaped under the hood.
  • Collision risk — implementation and proxy must agree on storage slots (incl. EIP-1967 reserved slots), packed structs, and keccak256(slot) + n mapping layout.

ABI encoding

  • Function selector: first 4 bytes of keccak256(canonical_signature) — e.g. transfer(address,uint256).
  • Dynamic types use head/tail (offset + payload) layouts in calldata.

Events / logs

  • Logs are not readable by contracts during execution in the obvious way; they are cheap off-chain observability signals. Execution headers carry a logs bloom for light-client filtering (legacy shape; roadmap work continues elsewhere).

Reentrancy (classic)

  • Cause: external call before state is settled.
  • Mitigations: checks-effects-interactions, reentrancy guards, transient locks (EIP-1153), pull payments.

Segment 4 — Rollups, DA, 4844, AA, consensus, tries, precompiles, MEV, Rust ecosystem, ROI

Rollups & data availability

  • Execution ≠ data availability. A rollup must publish (or inherit) enough data for others to reconstruct state transitions — otherwise it is not trust-minimized in the usual rollup story.
  • Optimistic rollups (e.g. Arbitrum, OP Mainnet) — fault/challenge windows for correctness arguments.
  • Validity / ZK-style systems — cryptographic proofs of correct execution; zkSync-style chains target EVM compatibility; Starknet is a major validity rollup built around the Cairo VM (not “plain EVM bytecode” — place it correctly in interviews).

EIP-4844 (proto-danksharding)

  • Introduced blob-carrying transactions: large blob sidecars committed with KZG commitments — not exposed as ordinary EVM calldata memory; proofs use the point-evaluation precompile path.
  • Separate blob gas market — multidimensional fee market (execution gas vs blob gas).
  • Blobs are short-lived on the consensus sidecar path (order of ~4096 epochs ≈ ~18 days of availability for reconstruction assumptions — verify exact parameter in the consensus spec you ship against).
  • Interview line: Ethereum is unbundling execution pricing from DA pricing for rollups.

Account abstraction

  • Classic model: EOAs sign txs. ERC-4337 adds an account abstraction mempool of UserOperations — bundlers, paymasters, validation hooks.
  • EIP-7702 (Pectra-era framing): EOAs can gain delegation to contract code for a transaction — “temporary smart account behavior” for EOAs in one-liner interviews; read the spec for exact semantics.

Consensus & finality (EL + CL)

  • Post-merge: execution + consensus layers; Gasper combines Casper FFG finality with LMD-GHOST fork choice.
  • Vocabulary: slots, epochs, validators, attestations, finalized checkpoints.
  • RLP — historic EL serialization; SSZ — consensus-layer typed tree hashing — both show up in Rust client codebases.

State trie

  • Patricia-style tries authenticate accounts and storage; transaction trie / receipt trie roots appear in block headers — enables proofs and light-client models.

Precompiles

  • Native at fixed addresses: ecrecover, sha256, modexp, alt_bn128 pairing checks, bls12-381 ops (fork-dependent), KZG point evaluation (4844-era) — used by signatures, bridges, and ZK verification paths.

MEV & PBS

  • MEV — value extractable by reordering / inserting txs.
  • Actors: searchers, builders, relays, validators — PBS (proposer–builder separation) makes block construction an auction-shaped pipeline in practice.
  • Patterns: sandwiching, arbitrage, liquidations, backruns — know the ethics and engineering mitigations at a high level.

Rust + Ethereum infra (interview flex, accurate)

  • High-impact Rust stacks: reth (execution client), revm (EVM interpreter), alloy (types / RPC), ethers-rs (still common in older code; many migrate toward alloy).
  • Paradigm is the team / org behind much of that Rust surface area — name libraries, not “Paradigm” as a crate.
  • Erigon is a major execution client but primarily Go — cite it as ecosystem breadth, not as a Rust flagship.
  • Why Rust here: performance, memory safety, and concurrency discipline for syncing / state — a defensible interview sentence without claiming exclusivity.

Rust-specific angles (if they go deep)

Async vs deterministic replay, state DB caching, trie I/O, zero-copy decoding, mempool design, speculative / parallel execution research vs canonical serial replay, interpreter vs JIT — tie each back to determinism and same gas semantics.

Highest-ROI topics (if you only pick five)

  1. Rollups + DA
  2. EIP-4844 + blob gas
  3. DELEGATECALL + proxy storage discipline
  4. MEV + PBS
  5. Storage / gas economics

One-liners that land (use sparingly, mean them)

  • “State growth dominates many workloads more than raw ALU.”
  • “Rollups move bulk execution off L1; L1 specializes in security, settlement, and DA commitments.”
  • “4844 split execution gas from blob data gas.”
  • “Delegatecall decouples who owns storage from who authored the bytecode.”
