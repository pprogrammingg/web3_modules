0 — Prerequisite mental model

  • Shard = validator committee (ShardGroupId in code). Example config only: e.g. 26 shards × 50–100 validators per shard—not fixed by the protocol.
  • Validator = one process: IoLoop + NodeStateMachine (BFT + mempool + execution) + storage + network impl.
  • Mempool = per-validator in-memory state (hyperscale-mempool), not one shared shard mailbox.
  • Production network = hyperscale-network-libp2p (libp2p gossipsub over QUIC between validators—separate from the wallet’s HTTP POST in step 2). Simulation uses hyperscale-network-memory — noted only where it differs.

0b — Why gossip and a local mempool?

Read this before steps 3–9. RPC (step 2) only hits one validator; gossip and mempool answer “how do the rest of the committee get the tx, and what happens on this machine before block building?”

Why gossip is necessary

  • Replication, not “remote mempool write.” Validators do not pull txs from a shared shard mailbox. Gossip copies the same signed bytes to peers on libp2p so each committee member can validate and decide locally.
  • Censorship resistance. A single RPC endpoint can drop you; gossip from any honest ingress is enough for the bytes to spread (Phase 1 staff debate on multi-submit).
  • Cross-shard fanout. A tx’s declared reads/writes may touch several shards; the first receiver gossips to each involved shard’s topic so those committees see the tx too (Phase 4).
  • Matches the wire protocol. Production uses gossipsub (many small validator messages), not N separate HTTP posts from the wallet to every node.

What the mempool is for (and what it is not)

The mempool is not the crossbeam queue, not “the network inbox,” and not where RPC writes when you POST. Those are ingress/scheduling layers (steps 3–5).

Role after validation (steps 8–9): on each validator, MempoolCoordinator holds txs that are cryptographically valid and policy-eligible as Pending candidates. That pool is the local staging area for consensus:

  • The proposer (Phase 2) pulls from ready_transactions() when building the next block—not from RPC or from “the shard’s global pool.”
  • Non-proposers keep the same tx Pending so they can vote on a block that includes it, track expiry/tombstones, and stay consistent with peers.
  • A validator on a shard the tx does not touch may still gossip and validate for relay, but skips local admission (involves_local_shard)—no entry in its pool.

So: gossip = spread bytes; mempool = per-validator “ready to be proposed/voted on” set once this node trusts the signature and Radix rules.

Why “batch” keeps appearing (three different things)

“Batch” in Phase 1 is not “the IoLoop dequeues many crossbeam messages in one turn.” Production still does one NodeInput → one step per loop iteration (step 4).

1 — Production pipeline (numbered)

Proof snippets below are trimmed from the hyperscale-rs tree. Ellipsis (...) hides boilerplate; follow data-file links for full sources.

1

User signs on device

On the phone or browser, a wallet (or dApp) builds a Radix NotarizedTransactionV1: the transaction intent plus a user signature that proves the account owner authorized it. That user signature uses Ed25519, which is the normal Radix end-user signing scheme—fast, one signature per transaction, verified later by TransactionValidation in the node.

This is not the same cryptography validators use among themselves. Committee votes and quorum certificates use BLS (aggregatable signatures for many validators). Confusing the two is a common onboarding mistake: Phase 1 follows the user tx bytes; Phase 2 follows validator BLS votes and QCs. See also cryptography module for the full split.

2

HTTP RPC to a validator

The wallet submits the signed transaction to a validator using ordinary HTTP, not the validator-to-validator P2P stack. In production the usual entrypoint is POST /api/v1/transactions on the node’s RPC server (implemented in the production crate). The body carries the transaction as hex or SBOR bytes—the wire encoding the RPC handler decodes before any consensus logic runs.

A phone or browser is not expected to open a libp2p connection or publish to gossipsub topics. Wallets talk HTTP to one (or a few) validators they trust for submission; after the RPC handler accepts the payload, the node forwards work internally. Validator replicas still meet on libp2p later in this phase—that is how peers share the same tx bytes, not how the user’s app first delivers them.

3

RPC handler → crossbeam

On a thread that serves HTTP (alongside Tokio and libp2p), the RPC handler decodes the body (hex or SBOR → a typed transaction struct) and may reject early with backpressure—for example if the node is far behind on sync, the mempool snapshot reports pressure, or cross-shard congestion limits apply.

Not yet full cryptographic validation: this step does not prove the tx was signed by a valid account key. The handler does not call mempool admission and typically does not run the full TransactionValidation path (Ed25519 + Radix manifest rules) on the RPC thread—that happens later on a worker pool (Track B below). RPC may still return an error for obviously malformed payloads; soundness for “signed by the right private key” is established only after TransactionValidated (step 8).

When the handler accepts the bytes, it enqueues NodeInput::SubmitTransaction { tx } on a crossbeam channel (tx_submission_tx → consensus_rx in runner.rs) so the pinned consensus thread can process it safely.

In-repo proof — decode + crossbeam only; comments name the async path (no mempool here)

// crates/production/src/rpc/handlers.rs — submit_transaction_handler
if let Some(rejection) = check_backpressure(&state) { return rejection; }
let transaction = match decode_transaction(&request.transaction_hex) { ... };
// Submit directly to IoLoop via crossbeam channel.
// IoLoop will: 1. Gossip  2. Queue batch validation  3. Mempool after validation
state.tx_submission_tx.send(NodeInput::SubmitTransaction { tx: tx_arc }) ...;
// Return immediately - validation and gossip happen async
4

Pinned IoLoop receives

The production runner’s run_pinned_loop runs consensus on a dedicated pinned thread (io-loop). Other parts of the process—Tokio (HTTP RPC, libp2p, timer sleeps), worker pools (tx validation, heavy crypto)—must not call IoLoop::step directly. They hand work to that thread through several crossbeam channels, not one mixed inbox:

  • timer_rx — logical timers (proposal / view-change) armed on Tokio; when a sleep fires, it injects a timer-scoped NodeInput here.
  • callback_rx — results from off-thread work (e.g. validation finished on a DispatchPool) packaged as NodeInput for the hot path.
  • consensus_rx — “main” ingress: RPC SubmitTransaction (tx_submission_tx), libp2p-delivered gossip/BFT messages, and similar protocol inputs.

Why multiple channels? So the pinned loop can enforce priority without reordering a single FIFO: timer before callback before consensus on each receive. That favors liveness (a fired proposal timer is not stuck behind a gossip flood on consensus_rx). Separate queues also match who produces each message (Tokio timers vs pool callbacks vs network/RPC) and make queue-depth metrics meaningful. See also E2E tests: tracing code to flow for the full run_pinned_loop loop body.

One loop iteration vs one step: Each iteration of run_pinned_loop dequeues at most one NodeInput—priority try_recv on those three receivers, or a blocking crossbeam::select! if nothing is ready yet—then runs exactly one IoLoop::step(event) for that input. It does not drain every ready message from all channels in the same iteration. If a timer, a callback, and SubmitTransaction are all queued, this iteration takes the timer; the others wait for later iterations (often back-to-back, but still one step per iteration).

Do not confuse with “batch” in §0b: one crossbeam dequeue still means one step. “Batch” elsewhere means grouping many transactions for one gossip publish or one validation worker job—not merging unrelated NodeInputs.

In-repo proof — one event per loop iteration; timer > callback > consensus

// crates/production/src/runner.rs — run_pinned_loop
// Drains the three crossbeam channels via try_recv in priority order
// (timer_rx > callback_rx > consensus_rx). ...
if let Ok(e) = config.timer_rx.try_recv() { break 'recv Some(e); }
if let Ok(e) = config.callback_rx.try_recv() { break 'recv Some(e); }
if let Ok(e) = config.consensus_rx.try_recv() { break 'recv Some(e); }
// ...
if let Some(event) = event {
    let output = io_loop.step(event);  // exactly one NodeInput per iteration
    for op in output.timer_ops { timer_mgr.process_op(op); }
}
io_loop.flush_expired_batches(wall_clock_local());
5

step(SubmitTransaction)

One step(SubmitTransaction) on the pinned thread schedules two independent tracks in tx_validation.rs—replication to peers and local validation (see §0b for why both exist and what “gossip batch” means). Neither finishes inside this single step; both complete in later loop iterations.

Track A — schedule gossip (step 6 publishes). From the tx’s declared read/write node IDs, the node derives every involved shard and calls enqueue_tx_for_gossip—append to that shard’s outbound BatchAccumulator, not an immediate libp2p send per tx. The wire publish happens in step 6 when the accumulator hits count/time limits or flush_expired_batches runs.

Track B — validate, then maybe admit (steps 8–9). If this hash is not already in flight: queue_validation → BatchAccumulator → on flush, DispatchPool::TxValidation runs TransactionValidation (Ed25519 + Radix rules). Fail: TransactionValidationsFailed on callback_rx — no mempool. Pass: TransactionValidated on callback_rx → later step → admit_internal if involves_local_shard.

Gossip ingress (step 7): same queue_validation path as RPC submit — peers do not skip crypto because the tx was relayed.

Track A + B together (incl. multi-shard): One step(SubmitTransaction) can schedule gossip to every involved shard and local validation at once. Gossip (step 6) is not rolled back if local validation later fails — bytes may already be on the wire; invalid txs simply never reach Pending here or on peers after they validate.

Validation pass on this validator: worker OK → TransactionValidated → admit_internal only if this node’s shard is in the tx’s shard set; otherwise validate-and-relay only (no local pool). Other shards’ validators received gossip (step 6–7), run the same pipeline, and admit on their shard if involved. Cross-shard execution is Phase 4; inclusion in a block is Phase 2.

Validation fail on this validator: TransactionValidationsFailed — no mempool, no Phase 2 path for this hash. Gossip already sent is harmless noise; peers reject the same bytes in their own TransactionValidation and also skip admission.

In-repo proof — Track A + B in one handler; validation async via event_sender

// crates/node/src/io_loop/step/tx_validation.rs — handle_submit_transaction
for shard in shards {  // reads ∪ writes → shard_for_node

}
if !self.pending_validation.contains(&tx_hash) && !self.caches.tx_store.contains(&tx_hash) {
    self.locally_submitted.insert(tx_hash);
    self.pending_validation.insert(tx_hash);
    self.queue_validation(tx);
}

// flush_validation_batch — DispatchPool::TxValidation (off pinned thread)
self.dispatch.spawn(DispatchPool::TxValidation, move || {

        .map(|tx| validator.validate_transaction(tx).is_ok()).collect();
    if valid {
        let _ = event_tx.send(NodeInput::TransactionValidated { tx });
    } else {
        let _ = event_tx.send(NodeInput::TransactionValidationsFailed { hashes: ... });
    }
});

Gossip publish — batched per shard; may flush on cap or deadline

self.network.broadcast_to_shard(shard, &gossip);
6

Gossip leaves this validator

When a TransactionGossip batch is ready (queued in step 5, flushed by the network layer or batch timing), the production network stack publishes it to other validators on each involved shard. Under the hood that is libp2p gossipsub over a QUIC transport—validator-to-validator replication, separate from the wallet’s HTTP POST in step 2.

The purpose of gossip here is to copy the same signed transaction bytes to the rest of the committee so each peer can validate and, if appropriate, admit the tx into its own mempool. A common misconception is that gossip “writes into the destination mempool” or that RPC somehow updates remote pools directly. It does neither: every validator still runs validation and admit_internal locally after it receives TransactionGossipReceived (step 7).

In-repo proof — libp2p handler forwards bytes; mempool unchanged here

// crates/node/src/io_loop/network_handlers.rs
self.network.register_gossip_handler::<TransactionGossip>(TopicScope::Shard,
    move |gossip: TransactionGossip| -> GossipVerdict {
        for transaction in gossip.transactions {
            let _ = tx.send(NodeInput::TransactionGossipReceived { tx: transaction });
        }
        GossipVerdict::Accept
    },
);
7

Peers: gossip → validate

When another validator receives the gossip, libp2p delivers NodeInput::TransactionGossipReceived into its IoLoop, which feeds the same batched validation pipeline used for locally submitted transactions—unless the hash is already in TxStore or tombstoned (terminal, must not re-admit). Peers do not skip signature or Radix checks just because the tx originated elsewhere.

In-repo proof — gossip ingress uses same queue_validation as submit

// crates/node/src/io_loop/step/tx_validation.rs — handle_gossip_received_tx_for_validation
if !self.caches.tx_store.contains(&tx_hash) && !self.state.mempool().is_tombstoned(&tx_hash) {
    self.pending_validation.insert(tx_hash);
    self.queue_validation(tx);
}
8

TransactionValidated

This step is the first time the state machine treats the tx as cryptographically sound—Ed25519 and Radix checks already succeeded in TransactionValidation on a worker (Track B). The pinned thread now raises ProtocolEvent::TransactionValidated into NodeStateMachine. The mempool coordinator admits the transaction only if this validator’s topology says the tx involves the local shard (topology.involves_local_shard(&tx)). Validators on shards the tx does not touch may still have gossiped or validated the bytes for relay, but they will not keep it in their local pool.

In-repo proof — worker success → feed_event(TransactionValidated)

// crates/node/src/io_loop/step/tx_validation.rs — handle_transaction_validated
self.feed_event(ProtocolEvent::TransactionValidated { tx, submitted_locally });

// crates/node/src/state/transactions.rs — on_transaction_validated
if !self.topology.snapshot().involves_local_shard(&tx) {
    return vec![];  // validate-and-relay only
}
self.mempool.on_transaction_gossip(...)
9

Mempool admission → Pending

After TransactionValidated, the mempool coordinator’s admit_internal routine decides whether this validator keeps the transaction in its local pool (§0b: staging for Phase 2 proposer selection—not ingress). It is the gate between “cryptographically valid” and “eligible to be proposed in a future block.”

Typical reject reasons:

  • Tombstone — the hash was already finalized or evicted and must not be re-admitted.
  • Duplicate — the same hash is already in the pool with a live status.
  • Expired — the transaction’s validity_range.end is before the node’s current chain time (WeightedTimestamp from committed QCs, not the machine’s wall clock).

If admission succeeds, the tx is inserted as Pending in this validator’s BTreeMap<TxHash, PoolEntry> (with ready/deferred tracking for proposer selection later). New pool content can set a latch via queue_ready_proposal(), which tells BFT that fresh material exists—but building and voting on a block still happens in Phase 2, not in this step.

In-repo proof — admission gate is admit_internal; success emits TransactionsAdmitted

// crates/mempool/src/coordinator.rs — on_transaction_gossip
match self.admit_internal(topology, &tx, submitted_locally, now) {
    Some(_) => vec![Action::Continuation(ProtocolEvent::TransactionsAdmitted {
        txs: vec![tx],
    })],
    None => vec![],
}

Single-shard outcome: For a transaction that only touches one shard, every validator in that shard’s committee that receives and successfully validates the gossip should end with the same hash in Pending inside its own BTreeMap<TxHash, PoolEntry>—there is still no shared global mempool object. A future proposer on that shard will later choose from ready_transactions(); inclusion in a block is not part of Phase 1.

3 — Design Q&A (curated)

Is there a distinct “RPC client” in the protocol?

No. Any HTTP client can POST. Validators are servers; wallets are clients. Validators also gossip on libp2p to each other.

Where does the validator verify the user Ed25519 signature?

On a DispatchPool::TxValidation worker, after queue_validation / batch flush—not on the HTTP RPC thread and not inside admit_internal.

Flow: RPC decodes and enqueues SubmitTransaction (step 3) → pinned step(SubmitTransaction) schedules validation (step 5) → worker runs TransactionValidation → success returns as TransactionValidated on callback_rx (step 8) → only then mempool admission (step 9). Forged or wrongly signed txs fail in validation; they never reach Pending.

Validator BLS signatures (votes, QCs) are a different path—Phase 2, not user tx ingress.

Why crossbeam instead of pushing straight into the mempool?

Because the mempool is not a public inbox that any thread can append to. On each validator, MempoolCoordinator lives inside NodeStateMachine and is updated only when the pinned IoLoop thread runs step on a NodeInput or ProtocolEvent. That design keeps BFT, mempool, and execution state in one place without mutexes on every hot path.

The HTTP RPC handler runs on a different thread (Tokio/async world alongside libp2p and validation workers). When a wallet POSTs a transaction, the handler can decode the body and apply backpressure, but it must not call admit_internal directly—that would race with the consensus thread mutating the same maps.

Instead the handler sends a small message on a crossbeam channel: NodeInput::SubmitTransaction { tx }. The IoLoop thread receives it, runs step(SubmitTransaction), gossips and schedules validation, and only later—after TransactionValidated—does the state machine call mempool admission. So the channel is a thread-safe handoff of work, not “the mempool queue.” The actual pool is still updated synchronously on the owner thread, in the right order with gossip and validation.

If you imagined “RPC writes straight into mempool,” you would need either heavy locking around the whole pool (slow, easy to get wrong) or a second code path that bypasses validation ordering. Hyperscale-rs avoids both by funneling everything through NodeInput → step → events.

Why several crossbeam channels instead of one Sender<NodeInput>?

One FIFO queue would force you to either process strictly in arrival order (bad for liveness when gossip bursts ahead of a fired proposal timer) or scan/reorder on every dequeue. Production splits producers into timer_rx, callback_rx, consensus_rx so run_pinned_loop can apply a fixed policy: timer → callback → consensus on each receive.

That matches how work is created: Tokio timer tasks, validation/execution pools, and network/RPC tasks all run elsewhere; each sends a small NodeInput when ready. The pinned thread stays the sole mutator of BFT + mempool + execution orchestration.

What is an IoLoop::step?

One atomic turn of the validator’s orchestrator on the pinned io-loop thread. The production runner dequeues exactly one NodeInput from crossbeam (timer, callback, or consensus priority), then calls IoLoop::step(that_event) in io_loop/mod.rs.

Inside that single step, the loop does one of two things:

  • IoLoop-only path — handle the variant directly in step without entering the big state machine yet. Examples: SubmitTransaction (schedule gossip + validation), TransactionGossipReceived, TransactionValidated (then feed mempool admission).
  • State-machine path — wrap a ProtocolEvent (often via NodeInput::Protocol(...)) and call NodeStateMachine::handle, then run every returned Action (network publish, timers, storage, continuations).

step returns a small StepOutput (status emissions, timer ops, action counts). It does not drain the whole crossbeam queue. Housekeeping such as flush_expired_batches may run in the same outer run_pinned_loop iteration after step, but that is not a second NodeInput.

Mental model: one NodeInput = one “something happened” message; one step = “process that one thing and emit whatever the node must do next.”

Can one round of run_pinned_loop process multiple crossbeam events?

No—one outer-loop iteration handles one dequeued NodeInput and one IoLoop::step. After step, the same iteration may run post-step housekeeping (flush_expired_batches, metrics, optional JVT GC)—that is not consuming another channel message.

If many inputs are pending, the loop may spin through iterations quickly (timer, then callback, then consensus, …), but each iteration still advances the state machine with one logical event. Throughput comes from fast step handlers and worker pools, not from batching unrelated NodeInputs into a single step call.

Does “gossip batch” mean crossbeam delivers many txs in one IoLoop turn?

No. Crossbeam still delivers one NodeInput per loop iteration. “Gossip batch” means many transactions are packed into one outbound TransactionGossip message per shard via BatchAccumulator (count cap + time window), so libp2p is not flooded with one publish per tx. A separate validation batch groups txs for one worker job—see §0b table.

If the tx only touches my shard, do we skip gossip?

No. Peers still need the bytes. You also validate locally → TransactionValidated without waiting for your own gossip echo.

Gotcha — “land in the destination mempool”: There is no global mempool for the whole blockchain and no shared mempool object per shard either. Each validator process keeps its own in-memory pool. Gossip only copies the signed transaction bytes to peers; after each validator validates locally, it runs the same admit_internal rules on its own MempoolCoordinator—so “arriving at the shard” really means “copied to every committee member who then admits independently.”

Can an attacker submit someone else’s signed tx?

Yes (replay/spam/propagation). Forgery fails TransactionValidation. Censorship: one RPC endpoint can drop traffic — mitigate with multi-submit and honest gossip; protocol needs quorum, not one gateway.

Do validators gossip transactions over libp2p on QUIC in production?

Yes, in production. After a validator packages a TransactionGossip message, the network-libp2p stack publishes it to peers using libp2p gossipsub on a QUIC transport. That is validator-to-validator replication on shard-scoped topics—not the wallet’s HTTP RPC path from step 2.

In simulation and tests, the same message types flow through an in-memory network adapter (fake latency queues, no real QUIC handshake on the validator mesh). The protocol logic and SBOR payloads are the same; only the transport layer changes. When you read traces, “gossip left the node” in prod means gossipsub/QUIC between validators; in sim it means the harness delivered bytes to peers on a deterministic schedule. Neither path is the wallet’s HTTP submit from step 2.

4 — Staff debate: pros, cons, mitigations

RPC to one validator (default UX)

Wallets and dApps can use a single HTTPS URL—easy to document and for users (“send tx to this RPC”).

How mitigation works in practice: the wallet sends the same already-signed bytes to more than one validator RPC (e.g. three endpoints run by different operators). Each POST /api/v1/transactions is independent. If validator A drops your request, B or C may still accept it, run SubmitTransaction, and gossip to the shard. You only need one honest ingress; gossip spreads the tx to the rest of the committee.

Typical patterns: a primary plus fallback RPC list in the wallet; parallel background POSTs; an app backend that fans out; or public infra that load-shares across operators. No resigning is required—the signature is already on the payload.

What can still go wrong

One URL only: if the user never leaves a censoring RPC, that node never helps—but another endpoint can, without a new signature from the user.

Captured endpoint list: if every RPC URL the wallet knows is run by colluding operators who all drop the same tx, ingress fails. That is curated trust in RPC providers, not a protocol-level fix.

Pinned single-threaded IoLoop

No locks on BFT/mempool; clear happens-before.

Cost

All ingress is serialized on one thread: each loop iteration runs one step, so a slow handler delays the next dequeue (timer, callback, or consensus).

Mitigation: Heavy work on worker pools; completion returns as NodeInput on callback_rx. Validation batches flush on deadlines without merging multiple unrelated inputs into one step.

Gossip even for same-shard submit

Peers stay consistent; matches pubsub reality.

Cost

Bandwidth vs hypothetical direct fanout RPC.

Mitigation: Batched TransactionGossip; receiver dedup.

Per-validator mempool

No central shard bottleneck; parallel admission.

Cost

Brief drift until gossip converges.

Mitigation: Same rules + eventual inclusion in a block.

5 — Phase 1 checklist

  • User signed a Radix NotarizedTransactionV1 with Ed25519 (wallet/dApp—not validator BLS)
  • HTTP POST /api/v1/transactions accepted (hex/SBOR body; optional: submit to several validators)
  • SubmitTransaction on crossbeam → pinned step
  • Gossip to every shard in the tx’s declared node set (single-shard = one shard)
  • Validate → TransactionValidated
  • MempoolCoordinator: Pending on each admitting validator
  • Not yet: block proposed, committed, or executed → Phase 2
