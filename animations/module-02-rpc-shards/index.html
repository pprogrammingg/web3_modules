After mempool — what actually happens?

The mempool is local state on each full node behind that RPC (not global magic). When it’s time to move forward, shard routing decides which validator set “owns” the transaction (by account, address space, or whatever rule your chain uses).

It is usually not “any random shard node grabs one tx.” Under consensus, the proposer for that shard and round (leader election / round-robin in BFT) builds a candidate block and pulls eligible txs from that shard’s mempool; other validators on that shard only verify and vote. Everyone on the shard applies the same committed block — inclusion is rule-driven.

Where does the block end up?

The block’s first hop is inside the shard: gossip of the proposal / header to fellow validators on that shard, then votes, then a quorum certificate and commit onto that shard’s chain. It does not “fly to every other shard” as one copied block — other shards have their own proposers and blocks at the same logical time. Cross-shard txs use extra phases (provisions, execution) after each shard has committed what it must — see Cross-Shard / execution modules in the main course.
