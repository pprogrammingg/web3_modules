Hyperscale-rs Overview & Setup

⏱️ Duration: 1.5-2 hours 📊 Difficulty: Basic 🎯 Hyperscale-rs Specific

Learning Objectives

By the end of this module, you will:

  • Know what hyperscale-rs is and where to find the repo and docs
  • Understand the best ways to trace transaction flow end to end (run, debugger, logs)
  • Know what it takes to run real nodes (requirements, local cluster, Docker, or distributed) and how that differs from simulation
  • Know the next step: complete the Crate Groups module and its quizzes

1. Intro to the project: what it is and links

Hyperscale (the protocol)

Hyperscale is the BFT consensus protocol designed for high-throughput, sharded blockchain networks. It defines how validators agree on blocks, handle cross-shard coordination, and achieve finality. The protocol and the Rust implementation live in the same repository.

Hyperscale-rs (the implementation)

Hyperscale-rs is the Rust implementation: a Byzantine Fault Tolerant (BFT) consensus system for high-throughput, sharded networks. It is not a full blockchain platform — it is the consensus layer that decides which transactions get included and in what order. Execution, storage, and user-facing apps are separate.

Key links:

  • Repository — code and protocol docs
  • README.md — setup and “What’s different”
  • StateMachine trait, node state — core types to skim early

Design: pure consensus (no I/O in core), state machine model, event-driven (events in → actions out), deterministic so the same code runs in simulation and production.

2. Trace flows end-to-end

The system is "events in, actions out": the node receives Events and returns Actions; the runner does I/O (network, storage). Below are three flows with step-by-step instructions: where to add tracing::debug!, the command to run the sim, and the log order you should see. Do the prerequisite once, then pick one flow at a time.

Prerequisite: Build and run (do this first)

  1. Clone: git clone --recurse-submodules https://github.com/flightofthefox/hyperscale-rs.git, then cd hyperscale-rs. If you already cloned without submodules: git submodule update --init --recursive.
  2. Build: cargo build --release. Wait until it finishes without errors.

Tracing: Use tracing::debug! (or debug! with use tracing::debug). Run the sim with RUST_LOG=debug so these logs show up. Remove or comment out the lines before committing.

Flow 1: Transaction submit (workload → sim runner → node → mempool)

Overview: The simulator workload submits a transaction via schedule_initial_event(..., NodeInput::SubmitTransaction { tx }); the node's io_loop receives it and gossips to shards; the protocol layer delivers TransactionGossipReceived to the state machine and the mempool adds the tx. You see the path from submit to tx in mempool.

1. Where to add debug!

StepFilePlaceAdd
1 crates/simulator/src/runner.rs Inside the for tx in batch loop, right before the loop that calls schedule_initial_event for each node (around line 179), after you have tx and target_shard (e.g. after let tx = std::sync::Arc::new(tx);). tracing::debug!(tx_hash = ?hash, target_shard = target_shard.0, nodes = ?shard_nodes, "sim: workload submitting tx to shard nodes");
2 crates/node/src/io_loop/mod.rs In crates/node/src/io_loop/mod.rs, inside the NodeInput::SubmitTransaction { tx } arm (around line 492), at the top of the block (where shards are computed and gossip is broadcast). tracing::debug!(tx_hash = ?tx_hash, shards = ?shards, "io_loop: SubmitTransaction received, gossiping to shards");
3 crates/node/src/protocol/mod.rs Where Event::SubmitTransaction is handled (protocol layer). tracing::debug!(tx_hash = ?tx_hash, "protocol: SubmitTransaction intercepted, gossip + queue validation");
4 crates/node/src/state.rs At the start of the ProtocolEvent::TransactionGossipReceived { tx, submitted_locally } arm in the state machine match (around line 1035), before on_transaction_gossip_received is called. tracing::debug!(tx_hash = ?tx.hash(), submitted_locally = submitted_locally, "node state: TransactionGossipReceived -> mempool");
5 crates/mempool/src/state.rs In on_transaction_gossip_arc, right after let hash = tx.hash(); (around line 398). (There may already be a tracing::debug! a bit lower; ensure RUST_LOG=debug shows it, or add this.) tracing::debug!(tx_hash = ?hash, submitted_locally, "mempool: adding tx via gossip");

2. Command to run (tx submit in simulation)

RUST_LOG=debug cargo run --release --bin hyperscale-sim -- --seed 42 -s 1 -v 4 -d 5 --tps 20
  • RUST_LOG=debug — your debug! and existing debug logs appear.
  • --seed 42 — reproducible run.
  • -s 1 — one shard (simplest path).
  • -v 4 — 4 validators (BFT quorum).
  • -d 5 — 5 seconds.
  • --tps 20 — low TPS so logs are easier to follow.

3. Optional shorter run

To see just a few submissions: RUST_LOG=debug cargo run --release --bin hyperscale-sim -- --seed 42 -s 1 -v 4 -d 2 --tps 5 (2 seconds, 5 TPS).

Flow 2: Block vote (message in → BFT → vote out)

Overview: A vote for a block arrives from the network (or the node produces its own vote). The simulation runner decodes it and feeds an event into the node; the BFT layer processes it and may emit Action::SendMessage (vote); the runner sends it over the simulated network. You see the consensus vote round-trip.

1. Where to add debug!

StepFilePlaceAdd
1 crates/simulation/src/runner.rs Where incoming BFT messages are decoded and turned into an event for the node (search for the BFT message handler / event injection). At the start of that block. tracing::debug!(node = node_index, "sim runner: BFT message received, forwarding to node");
2 crates/node/src/state.rs At the start of the match arm that handles the BFT event (e.g. Event::Bft(...) or similar — the arm that processes votes). tracing::debug!("node state: BFT event handled (vote in/out)");
3 crates/bft or crates/node Where the node emits Action::SendMessage for a vote (e.g. in the BFT state machine or in the node's BFT handler when it decides to send a vote). tracing::debug!("bft: emitting SendMessage (vote)");
4 crates/simulation/src/runner.rs Where Action::SendMessage is executed (action dispatch / send to network). At the start of that branch. tracing::debug!(node = node_index, "sim runner: SendMessage executed (vote on wire)");

2. Command to run

Same as Flow 1 so that proposals and votes happen: RUST_LOG=debug cargo run --release --bin hyperscale-sim -- --seed 42 -s 1 -v 4 -d 5 --tps 20. Votes occur as the sim runs; watch for your four log lines in order.

Flow 3: Block commit (commit → execution)

Overview: BFT agrees on a block and commits it. The node receives a "block committed" event; the state machine hands off to the execution layer, which runs the transactions in that block. You see consensus output becoming execution input.

1. Where to add debug!

StepFilePlaceAdd
1 crates/node/src/state.rs At the start of the match arm that handles the block-committed event (e.g. Event::BlockCommitted or similar — search for "BlockCommitted" or "block committed" in this file). tracing::debug!(height = ?height, "node state: BlockCommitted, entering execution"); (use the height/block id variable that exists there if available, else drop the field.)
2 crates/execution or crates/node Where execution is invoked for a committed block (e.g. "execute block" or "apply block" entry point). At the start of that path. tracing::debug!("execution: applying committed block");

2. Command to run

Same as Flow 1: RUST_LOG=debug cargo run --release --bin hyperscale-sim -- --seed 42 -s 1 -v 4 -d 5 --tps 20. Blocks commit as the sim runs; you should see "BlockCommitted, entering execution" then "execution: applying committed block" when a block commits.

Understanding simulator output

When you run the sim, you may see metrics like these. Hover the terms for definitions:

  • Lock contention (peak) / Locked nodes — Locked nodes (peak) is the maximum number of distinct state nodes (accounts, components) held by in-flight transactions at any moment. Higher = more contention.
  • In-flight — In-flight transactions are submitted but not yet in a final state (completed, rejected, retried) at the time the run stopped. “In-flight: X (at cutoff)” = X txs were still in progress when the sim ended.
  • Account usage – Skew ratio — Skew ratio = max_selections / avg_selections across accounts used by the workload. 1.0 = perfectly even; higher (e.g. 2.50 or 12.50) means a few accounts were used much more than others.

Full picture: The complete end-to-end flow — from user signing a transaction to finality — is covered in a dedicated Basic module: Transaction Flow: User to Finality. Use that module for the full diagram and to map every step to crates.

3. Running real nodes (production) vs simulation

The rest of this module uses the simulator (hyperscale-sim): one process, simulated time, in-memory network. To run real validator nodes (separate processes, real libp2p, real RocksDB, real RPC), you use the production runner and launch scripts. Below: what it takes and how to do it.

Requirements

  • Build: cargo build --release (same as for sim). This builds the hyperscale-validator (or equivalent production binary) and the launch scripts use it.
  • Topology: Number of shards and validators per shard. The launch scripts generate validator keys and configs for you.
  • Network: Each node needs a libp2p listen address and (for distributed) reachable IPs/ports. Ports (e.g. 9000 UDP/TCP for P2P, 8080 for RPC) must be open on the host or between machines.
  • Storage: Each validator writes RocksDB data to a directory; the scripts create these or use --clean to start fresh.
  • Genesis: The cluster needs a shared genesis (validator set, etc.). The launch scripts generate this when they generate configs.
  • Docker (optional): If you use the Docker-based cluster, Docker (and Docker Compose) must be installed. Allocate at least 8–10 GB RAM to the Docker VM/Desktop so validators have enough memory.

How to run a local cluster (one machine)

Process-based (fastest for iteration):

./scripts/launch-cluster.sh

This starts a cluster of validator nodes as background processes on your host. Options:

  • --shards <N> — Number of shards (default: 2).
  • --validators-per-shard <M> — Validators per shard (default: 4).
  • --clean — Wipe data directories before starting.
  • --monitoring — Start Prometheus and Grafana for metrics.

At the end, the script prints the RPC ports for each node (e.g. 8080, 8081, …). You need these for the spammer or for submitting transactions.

Docker-based (closer to production):

./scripts/launch-docker-compose.sh

Runs the cluster inside Docker containers. Ensure Docker has at least 8–10 GB RAM. Options include --shards, --validators-per-shard, --memory, --cpus, --latency. Stop with:

./scripts/stop-docker-compose.sh

How to run a distributed cluster (multiple machines)

To run validators across several physical hosts or VMs, use the distributed workflow (see the repo’s README_DISTRIBUTED.md):

  1. Generate configs on one machine: ./scripts/generate-distributed-config.sh --hosts "IP1,IP2,..." --nodes-per-host N. This creates keys and config.toml per node under distributed-cluster-data/.
  2. Copy the hyperscale-validator binary and each host’s config directory to the corresponding machine.
  3. Start each validator on its host: ./hyperscale-validator --config /path/to/node-X/config.toml.

Prerequisites: Rust (for building and generating configs), SSH to all hosts, and ports 9000 (UDP/TCP) and 8080 (TCP) open as needed for P2P and RPC.

Load testing real nodes

After the cluster is up, use the hyperscale-spammer to send traffic to the RPC endpoints. Use the ports printed by the launch script (they depend on shards and validators):

./target/release/hyperscale-spammer run \
  --endpoints "http://localhost:8080,http://localhost:8081" \
  --num-shards 2 \
  --validators-per-shard 4 \
  --tps 100 \
  --duration 30s

Summary: Simulation = one binary (hyperscale-sim), no real network or disk. Real nodes = launch script or Docker/distributed + hyperscale-validator per node, real libp2p and RocksDB; then use the spammer or your own RPC client to drive load. For exact flags and troubleshooting, see the repo README.md and README_DISTRIBUTED.md.

Important crates for overall flows

When studying single-shard and multi-shard flows, these crates are the main ones to focus on:

  • Single-shard: node (state machine, protocol, event→action), mempool (tx pool, admission), bft (proposal, voting, QC, commit rule), execution (run committed block, StateVoteBlock → StateCertificate), types (Block, BlockHeader, QC, TransactionCertificate).
  • Multi-shard: Same as above, plus provisions (ProvisionCoordinator, StateProvision quorum, ProvisioningComplete), livelock (cycle detection, defer/retry), and execution’s cross-shard phases (provision verification, then execute, then certificate aggregation).

Simulation and tracing: simulator (workload, submit), simulation (runner, network, action dispatch). See Transaction Flow and Crate Groups for crate-by-crate quizzes and flow maps.
