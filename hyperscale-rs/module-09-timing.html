Timing: Rounds, Heights, Timeouts & Timers

⏱️ Duration: 1–1.5 hours 📊 Difficulty: Intermediate 🎯 Hyperscale-rs Specific

Learning Objectives

  • Understand height vs round (view) and how they drive BFT progress
  • Know how timeouts trigger round advancement (view change) when no quorum is reached in time
  • See how timers (Proposal, Cleanup) are set and fired in the node and BFT state machine
  • Map timing-related config (proposal_interval, min_block_interval) and state (view, view_at_height_start) to the codebase
  • Relate timing to liveness: why timeouts and view changes are necessary for progress under delay or faults

Why timing matters

BFT consensus must make progress even when the network is slow or some validators are slow or faulty. If the leader never proposes, or proposals never reach quorum, the chain would stall. Heights and rounds structure the protocol; timeouts and timers ensure that after some bounded time the system moves to a new round (new leader) or triggers the next proposal. This module goes into detail on how hyperscale-rs does that.

For the full BFT flow (propose → vote → QC → commit), see Dynamics of Node State Machine and BFT. Here we focus on the when: rounds, heights, timeouts, and timers.

Height and round (view)

Height is the index of the block slot in the chain (block 0 = genesis, block 1, 2, …). One height = one block to be decided. Round (also called view in the code) is the attempt number for that height. If round 0 doesn’t produce a quorum in time, the protocol advances to round 1 (new proposer, new proposal) for the same height, and so on until a QC is formed.

  • Height advances only when a block at that height gets a QC and (with the two-chain rule) is committed; then the next height starts.
  • Round advances on timeout: if no QC is formed before the round timeout, validators move to the next round (view change). The proposer for round r is typically committee[(height + round) % len] (round-robin), so each round has a different leader.

In BftState (see Suggested crates / files to study), the struct holds:

  • view: u64 — current round (view)
  • view_at_height_start: u64 — round when the current height started (used e.g. for backoff or logging)
  • committed_height: u64, committed_hash: Hash — chain progress

So: same height, multiple rounds = multiple attempts to get a block certified; height advances only when a block is committed (two-chain rule).

Timeouts and view change

If the proposer is slow, or the network delays messages, or not enough validators vote in time, the round may not produce a QC. To avoid waiting forever, the protocol uses a round timeout: if no QC is formed before the timeout expires, the round is abandoned and the system advances to the next round (view change).

  • Who checks the timeout? In the flow described in the BFT module, NodeStateMachine::handle() checks round timeout before dispatching Event::ProposalTimer to BFT. So the node layer decides “has this round run long enough?” and can advance the view or trigger the next proposal.
  • Effect: When the round times out, the next round starts: view increments (or is updated), a new proposer is chosen, and a new proposal can be made for the same height. Validators may then vote for the new proposal; that’s a revote (allowed), not equivocation (same height, different round).

Timeouts are what give the protocol liveness: eventually, either a QC is formed (progress at this height) or the round changes (new leader, new attempt). Under partial synchrony, after GST (global stabilization time), timeouts can be chosen so that progress is guaranteed.

Timers: Proposal and Cleanup

Hyperscale-rs uses timers to drive periodic actions. The BFT state machine doesn’t own a clock; instead it emits Action::SetTimer with a delay and a timer type. The runner (node/simulator) is responsible for firing the timer and enqueueing the corresponding event (e.g. Event::ProposalTimer).

Proposal timer

  • When set: After BftState::initialize_genesis(genesis), and again after BftState::on_proposal_timer(...) runs (so the next proposal is scheduled).
  • When fired: The runner delivers Event::ProposalTimer. The node’s NodeStateMachine::handle() runs: it checks round timeout, gathers ready_txs, certificates, etc., then calls self.bft.on_proposal_timer(ready_txs, deferred, aborted, certificates, commitment_proofs).
  • What happens: on_proposal_timer computes the next (height, round), checks if this node is the proposer for that (height, round); if yes, sets pending_proposal and returns Action::BuildProposal and another SetTimer for the next proposal tick. So the proposal timer drives “when do we try to propose the next block?”

Cleanup timer

  • When set: Returned together with the Proposal timer from initialize_genesis (and possibly elsewhere).
  • Purpose: Used to periodically clean old state (e.g. old pending blocks, expired vote sets) so memory and state don’t grow unbounded. When Event::CleanupTimer fires, BFT runs cleanup logic.

So: SetTimer is the way BFT asks the runner “wake me in Δ time”; the runner turns that into an event; BFT then reacts (propose, cleanup, or advance round depending on logic).

Config: proposal_interval and min_block_interval

BftConfig (and any node/simulator config that feeds it) typically includes:

  • proposal_interval — How often the proposal timer is set (e.g. every 500 ms). This drives how frequently the node attempts to propose (when it’s the proposer) or at least evaluates “should we propose?”.
  • min_block_interval — Minimum time between blocks (can be used to throttle block production or align with real time in simulation).

These values affect latency and throughput: shorter intervals mean faster rounds but more messages; longer intervals may reduce load but increase time to finality. Round timeout is usually derived or configured so that slow or missing proposals don’t block progress (view change after timeout).

For BftConfig and timer-related constants, use the bft crate and the node/simulator that constructs BftState — paths in Suggested crates / files to study.

Summary

  • Height = block index; round (view) = attempt for that height. Multiple rounds per height until a QC is formed; then height advances (after two-chain commit).
  • Timeouts ensure liveness: if no QC in time, round advances (view change), new proposer tries for the same height.
  • Timers: BFT emits SetTimer (Proposal, Cleanup); the runner fires events; on_proposal_timer drives “request block build” and schedules the next tick.
  • Config: proposal_interval, min_block_interval, and round timeout govern how fast the chain can progress and when view changes happen.
