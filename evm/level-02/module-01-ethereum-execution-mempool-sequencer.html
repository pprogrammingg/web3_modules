Ethereum execution, mempool & sequencer mental models

If you only know “high-throughput monolithic L1” patterns, L2 / rollup interviews will feel foreign. Segment 1 — Transaction lifecycle & mempool dynamics

End-to-end lifecycle (L1 mental model)

  1. Construct & sign — user or bot builds a typed transaction (legacy or EIP-1559 envelope): nonce, gas limits, to/data/value, chain id.
  2. Broadcast — tx enters the transaction pool of EL nodes (public gossip, private relays, builder networks—landscape is fragmented post-MEV).
  3. Ordering & inclusion — under proof-of-stake Ethereum, block builders (often via MEV-Boost-style flows) propose execution payloads; the validator signs the consensus block that carries that payload.
  4. Execution — EL runs the block: for each tx, deduct gas, run EVM, update accounts/storage, emit logs/receipts.
  5. Commitment — block header carries roots (state, transactions, receipts) that anchor what light clients and bridges believe about post-state.

Mempool dynamics (what interviewers probe)

  • Not a single global queue. Each node has local policy (fee thresholds, pool limits, replacement rules under EIP-1559).
  • Ordering is economic. Effective priority fee + block space + builder preferences determine inclusion, not FIFO fairness.
  • Private order flow — large share of volume never hits the public mempool; protocol engineers still need the interfaces (bundles, builders, relays) because they shape latency and reorg risk.
  • Pending ≠ final. Users and apps must reason about inclusion probability and reorg depth, not only “submitted.”

Sequencer-shaped systems (preview)

Rollups often add a sequencer that orders L2 txs before data is posted to L1. The mental model differs from L1:

  • Soft ordering — fast pre-confirmation from the sequencer feed (trust / policy).
  • Hard ordering — data actually available on L1 (calldata or blobs) so anyone can reproduce execution.

Segment 4 expands the security split; Arbitrum Nitro is a concrete instance.

Segment 2 — EVM execution pipeline, reorgs, state roots

EVM execution pipeline (inside one block)

For each transaction, conceptually:

  1. Intrinsic validation — signature, nonce, balance for max fee, chain rules.
  2. Gas metering — charge for calldata, access lists (EIP-2930), execution opcodes, memory expansion, storage warm/cold (EIP-2929), refunds capped per spec.
  3. Message call execution — top-level call into contract or transfer; internal CALL / DELEGATECALL tree with depth and stipend limits.
  4. State transition — update account balances, nonces, code hashes, storage trie; revert rolls back that tx’s changes.
  5. Receipt & logs — status, gas used, logs for indexers (not readable on-chain during execution).

Key insight: execution is deterministic given inputs—the hard part is agreeing on ordering and available data (DA), not opcode semantics.

Reorg implications

  • Post-merge Ethereum targets short, shallow reorgs under normal conditions—but never zero for apps at tip.
  • Reorg = chain switches to a fork with higher weight; txs from discarded blocks may return to mempool or disappear if underpriced.
  • Engineering responses: wait N confirmations, track blockHash in contract logic carefully (use block.number for logic, not trust in hash after EIP-2935 nuances), design bridges with challenge periods.
  • Rollups add another layer: L2 sequencer reorgs (rare policy-wise) vs L1 reorgs affecting batch inclusion.

State roots (what “settlement” points at)

  • World state trie — commitment to all accounts after block execution.
  • Receipts trie — per-tx outcomes.
  • Bridges and light clients care that the claimed L2 state root matches what you get by executing all inputs since genesis (or since a trusted checkpoint).

Segment 3 — Calldata, blobspace (EIP-4844), L2 settlement assumptions

Calldata costs (still matter)

  • Before blobs, rollups mostly paid L1 calldata gas for compressed batch bytes—often the dominant fee.
  • Calldata is ephemeral on L1 execution (not stored in state) but published in block data—consensus/EL clients must retain per policy.
  • Fallback paths still exist when blob markets are expensive or unavailable.

EIP-4844 blobspace

  • Proto-danksharding (4844) — separate blob gas market for large, ephemeral data with KZG commitments in the beacon chain.
  • Rollups post batch data as type-3 (blob-carrying) txs when economical; execution gas and blob gas are priced independently.
  • Interview line: “4844 decouples ‘execute on L1’ from ‘publish bulk data for L2 DA’—rollups optimize for blob fee, not only gas.”

L2 settlement assumptions (generic)

When an L2 says it “settles on Ethereum,” ask what is actually posted:

Posted to L1 What honest nodes need Typical security story
Batch data (calldata / blob) Inputs to replay L2 execution DA — anyone can reconstruct if they have the STF
State / output root claim Commitment to L2 result Validity (ZK proof) or fraud/challenge (optimistic)
Bridge messages Ordered inbox of L1→L2 or proofs for L2→L1 Asset safety separate from pure execution correctness

Data availability vs execution

  • Valid execution + missing data — useless for permissionless replay.
  • Data available + wrong execution — needs fraud proof, validity proof, or trusted prover.
  • Some L2s use committee DA (e.g. AnyTrust-style) — weaker than full L1 DA; know the trust model.

Segment 4 — Sequencer mental models & “inherit vs optimize”

Three clocks

  1. User clock — wallet sees RPC / sequencer feed confirmation.
  2. L2 clock — nodes execute ordered batches against L2 state.
  3. L1 clock — batches and assertions land on Ethereum; challenge windows run on L1 time.

What must inherit Ethereum security vs what can be optimized offchain

Topic Usually inherit / anchor on L1 Often optimized offchain
EVM opcode semantics Same rules if “EVM-equivalent” (modulo fork sync) Faster scheduling, parallel hints (implementation-specific)
Transaction ordering Eventually constrained by published batch order Sequencer feed, private mempool, Timeboost-style auctions
Data availability Rollup mode: blobs/calldata on L1 (or explicit alt-DA trust) Compression, batching, committee certs (AnyTrust)
Correctness of state transition Fraud/validity game on L1 (rollup-specific) Native execution speed, WASM/Stylus paths
Finality latency Hard finality tied to L1 inclusion + proof windows Soft finality in milliseconds–seconds
Censorship resistance Forced-inclusion queues (delayed inbox patterns) Happy-path sequencer RPC

Future/advanced pointers (read later)

  • PBS / MEV — proposer–builder separation on L1; analogs on L2 (e.g. ordering auctions).
  • Account abstraction (ERC-4337) — separate mempool for user operations; still settles through L1 execution eventually.
  • Statelessness / Verkle — changes witness costs, not the high-level sequencer story.

Interview one-liners

  • “Mempool is economic ordering, not FIFO—builders and private flow dominate at the tip.”
  • “Reorgs punish apps that treat pending as final.”
  • “4844 gave rollups a DA lane; execution gas is a different knob.”
  • “Sequencer soft finality is UX; L1-posted data + proofs are security.”
